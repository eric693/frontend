核心概念 01

DOM 是什麼？

瀏覽器把 HTML 解析成一棵樹。這棵樹就是 DOM——你用 JavaScript 讀取和操控頁面的唯一介面。

01
Document Object Model
HTML → 樹狀節點結構

DOM（Document Object Model，文件物件模型）是瀏覽器把 HTML 文件解析後所建立的樹狀物件結構。每個 HTML 標籤、文字內容、甚至注釋都成為一個節點（Node），節點之間有父子關係，形成一棵樹。

document 是整棵樹的根，也是你在 JavaScript 中存取所有頁面元素的起點。document.body、document.head、document.title 都是 DOM 提供的便捷入口。

理解 DOM 的關鍵概念：HTML 是靜態文字，而 DOM 是活的物件。當你用 JS 修改 DOM，頁面會立刻更新；但原始 HTML 檔案不會改變。

三種常見節點類型 Element Node（元素節點）：HTML 標籤，如 <div>、<p>。Text Node（文字節點）：標籤內的文字內容。Comment Node：HTML 注釋 <!-- -->。開發中最常操作的是 Element Node。

核心概念 02

查詢元素

操控 DOM 的第一步：找到你要操作的元素。現代 JS 幾乎只用兩個方法——querySelector 和 querySelectorAll。

02
querySelector vs querySelectorAll
單一元素 vs NodeList

document.querySelector(selector) 接受任何有效的 CSS 選擇器，回傳第一個符合的元素（或 null）。這讓你可以用第二週學的所有 CSS 選擇器技巧來精準定位元素。

document.querySelectorAll(selector) 回傳所有符合元素組成的 NodeList。NodeList 看起來像陣列但不完全是——它有 length 和可以 forEach，但沒有 map、filter 等陣列方法。若需要完整陣列功能，用 Array.from(nodeList) 轉換。

你也可以對已選取的元素呼叫 querySelector，限縮搜尋範圍：card.querySelector('.title') 只在 card 內部尋找 .title，而不是全域搜尋。

舊方法 vs 新方法 getElementById、getElementsByClassName、getElementsByTagName 是舊方法，雖然仍然有效，但現代開發幾乎都用 querySelector 取代。只需學會 querySelector 和 querySelectorAll，它們更靈活且支援所有 CSS 選擇器。

核心概念 03

修改 DOM 元素

選到元素之後，你可以修改它的文字、HTML、樣式、屬性和類別。每種修改方式有不同的適用場景和安全考量。

03
textContent vs innerHTML
安全性與功能的取捨

textContent 讀取或設定元素的純文字內容，會自動跳脫 HTML 標籤（把 < 轉成 &lt;），因此完全安全，不會被 XSS 攻擊。適合顯示使用者輸入的文字。

innerHTML 讀取或設定元素的HTML 字串，可以插入帶有標籤的 HTML。絕對不要把使用者輸入的內容直接用 innerHTML 插入——這是 XSS（跨站腳本攻擊）的主要來源。只在插入你自己控制的 HTML 模板時使用。

classList 提供管理 CSS 類別的乾淨 API：add()、remove()、toggle()（有就移除，沒有就加入）、contains()（檢查是否包含）。這是控制樣式狀態的標準做法，比直接修改 style 更好維護。

核心概念 04

事件監聽

事件讓頁面從「靜態展示」變成「動態互動」。每個使用者動作——點擊、輸入、滾動、按鍵——都是一個事件，你可以監聽它並執行對應的程式碼。

04
addEventListener
元素.addEventListener(事件類型, 回調函式)

addEventListener(type, handler) 是綁定事件的標準方式。type 是事件名稱（字串），handler 是事件觸發時要執行的函式。同一個元素可以綁定多個相同類型的事件，不會互相覆蓋。

回調函式會自動收到一個 event 物件（慣例命名為 e 或 event），包含事件的所有資訊：e.target（觸發事件的元素）、e.type（事件類型）、e.key（鍵盤事件的按鍵名稱）、e.clientX/Y（滑鼠位置）等。

避免使用舊的 HTML 行內事件屬性（onclick="..."）——它難以維護，且只能綁定一個處理函式。addEventListener 是現代標準做法。

核心概念 05

事件冒泡與事件委派

點擊頁面上的一個元素，這個事件會「向上冒泡」到所有父元素。理解這個機制，你就能用更少的監聽器處理更多元素，也能避免意外的行為。

05
事件冒泡（Bubbling）
從觸發元素向上傳播到 document

當你點擊一個元素，事件不只在那個元素上觸發——它會向上傳播（冒泡）到父元素、祖父元素，一路到 document。如果這些祖先元素也有相同事件的監聽器，它們也會觸發。

事件委派（Event Delegation）是善用冒泡的設計模式：把事件監聽器綁在父元素而非每個子元素上，透過 e.target 判斷是哪個子元素觸發的。這特別適合動態新增的元素——因為父元素的監聽器對後來加入的子元素同樣有效。

stopPropagation 的陷阱 e.stopPropagation() 停止事件繼續冒泡。但濫用它會讓程式碼難以追蹤——如果父元素的監聽器依賴冒泡機制，你的 stopPropagation 會悄悄破壞它。只在真正需要時才使用，大多數情況用 e.target 條件判斷更安全。

JavaScript — 事件委派與冒泡控制
// ── 問題：為每個 li 綁定監聽器（低效）──
document.querySelectorAll('li').forEach(li => {
  li.addEventListener('click', () => li.classList.toggle('done'));
});
// 問題：動態新增的 li 不會有監聽器！

// ── 解法：事件委派（Event Delegation）──
const list = document.querySelector('#todo-list');
list.addEventListener('click', (e) => {
  // e.target 是真正被點擊的元素
  if (e.target.tagName === 'LI') {
    e.target.classList.toggle('done');
  }
  // 更精準：使用 closest() 處理元素有子元素的情況
  const li = e.target.closest('li');
  if (li) li.classList.toggle('done');
});
// 優點：後來動態加入的 li 也會響應點擊

// ── 阻止預設行為 vs 阻止冒泡 ──
document.querySelector('a').addEventListener('click', (e) => {
  e.preventDefault();  // 阻止連結跳轉，不影響冒泡
  handleNavigation(e.target.href);
});

document.querySelector('.modal').addEventListener('click', (e) => {
  e.stopPropagation(); // 點 modal 內部不要觸發外層的關閉事件
});

// ── 常見模式：點擊外部關閉 ──
document.addEventListener('click', (e) => {
  const menu = document.querySelector('.dropdown');
  if (!menu.contains(e.target)) {
    menu.classList.remove('open'); // 點擊選單外部時關閉
  }
});
核心概念 06

動態建立與插入元素

不只是修改現有元素，你還可以從零建立新元素、插入到頁面任意位置，或移除不需要的元素——這是 Todo List、動態列表等互動功能的核心。

06
createElement vs innerHTML
兩種建立方式的取捨

document.createElement(tagName) 建立一個新元素，再用 append、prepend、insertBefore 等方法插入到 DOM。這種方式最安全（沒有 XSS 風險），也能直接對新元素綁定事件，適合需要精細控制的場景。

用 innerHTML 插入 HTML 字串則更簡潔快速，適合一次性插入複雜的 HTML 結構，或用 template literal 拼接資料。缺點是插入後需要再次查詢元素才能綁定事件，且有安全風險。

現代推薦做法 el.insertAdjacentHTML('beforeend', html) 比直接賦值給 innerHTML 更好——它不會清空現有內容，只是在指定位置插入。四個位置：beforebegin（元素之前）、afterbegin（子元素最前）、beforeend（子元素最後）、afterend（元素之後）。
createElement 方式（安全，可直接綁事件）
function addTodo(text) {
  const li = document.createElement('li');
  li.textContent = text; // 安全：無 XSS
  li.classList.add('todo-item');
  li.addEventListener('click', () =>
    li.classList.toggle('done')
  );
  list.append(li); // 插入最後
}
innerHTML 方式（簡潔，適合靜態模板）
function renderCard(item) {
  // ⚠️ item.name 不能是使用者輸入
  list.insertAdjacentHTML('beforeend', `
    <li class="todo-item">
      <span>${item.name}</span>
      <button class="del">✕</button>
    </li>
  `);
}

list.addEventListener('click', e => {

// ── 完整 Todo 範例（createElement + 事件委派）──
const list = document.querySelector('#list');

function addItem(text) {
  const li  = document.createElement('li');
  const span = document.createElement('span');
  const btn  = document.createElement('button');
  span.textContent = text;
  btn.textContent  = '刪除';
  btn.dataset.action = 'delete';
  li.append(span, btn);
  list.append(li);
}

// 事件委派：一個監聽器處理所有項目
list.addEventListener('click', (e) => {
  if (e.target.dataset.action === 'delete') {
    e.target.closest('li').remove();
  } else if (e.target.tagName === 'SPAN') {
    e.target.parentElement.classList.toggle('done');
  }
});

A兩者完全相同，都是觸發事件的元素
Be.target 是父元素，e.currentTarget 是被點擊的子元素
Ce.target 是實際觸發事件的最深層元素，e.currentTarget 是綁定事件監聽器的元素
De.target 只在 click 事件中有效，e.currentTarget 在所有事件都有效
CORRECT當你使用事件委派在父元素綁定監聽器時，e.currentTarget 永遠是那個父元素（監聽器所在之處），而 e.target 是使用者實際點擊到的最深層子元素。事件委派依賴的正是 e.target 來判斷「哪個子元素被觸發了」。
TRY AGAIN想像你在一個 <ul> 上綁定了 click 事件，使用者點擊了其中一個 <li>——此時監聽器在哪？事件實際發生在哪？
QUIZ 03 / 04
你有一個 Todo 列表，使用者可以動態新增項目。以下哪種做法讓新增的項目也能響應刪除點擊？
A每次新增後，重新用 querySelectorAll 選所有項目並重新綁定
B在 HTML 中用 onclick 屬性直接寫事件處理程式碼
C在每次 createElement 後立刻對新元素呼叫 addEventListener
D在父元素（ul）上用事件委派，透過 e.target.closest() 判斷點擊目標
CORRECT事件委派是最優的做法。在父元素綁定一個監聽器，利用冒泡機制捕捉所有子元素的點擊，用 e.target.closest('.delete-btn') 判斷是否點到刪除按鈕。這樣無論什麼時候新增的項目，都自動被同一個監聽器覆蓋，效能最好。
TRY AGAIN注意「動態新增的項目也能響應」這個條件。選項 C 雖然能讓當時新增的元素有監聽器，但也是可行的做法——只是事件委派更優雅。想想哪個方案一個監聽器就能搞定所有現在和未來的元素？
QUIZ 04 / 04
執行 document.querySelectorAll('.item') 後，你想對每個結果用 .map() 方法轉換。但直接呼叫報錯，原因是什麼？
AquerySelectorAll 回傳的是 NodeList，不是 Array，需要先用 Array.from() 或 [...nodeList] 轉換
B.map() 方法不能用在 DOM 元素上
C需要先呼叫 .toArray() 方法
DquerySelectorAll 只有在有結果時才能使用
CORRECTNodeList 是類陣列物件（array-like），有 length 屬性和索引，也支援 forEach，但沒有 map、filter、reduce 等陣列方法。轉換方式：Array.from(nodeList) 或展開運算子 [...nodeList]，兩者都能建立一個真正的陣列。
TRY AGAIN回想一下 querySelectorAll 的回傳型別——它不是 Array，而是一個特殊的「類陣列」物件。

動手做：互動式測驗 App

這週的專案要求你綜合運用 DOM 查詢、修改、事件監聽、動態建立元素，做出一個完整的多題測驗應用程式。

WEEK 05 PROJECT
互動式
測驗 App

純 HTML/CSS/JS 實作，不使用任何框架。包含題目顯示、選項點擊、分數計算、結果頁面，以及重新開始功能。

完成步驟

建立資料結構
建立一個 JS 陣列儲存題目，每道題是一個物件，包含：question（題目文字）、options（選項陣列，4 個字串）、answer（正確選項的索引，0–3）。至少準備 5 道題。
HTML 骨架與 CSS 樣式
建立基本 HTML 結構：#quiz-container（主容器）、#question-text（題目區）、#options-list（選項容器）、#progress（進度）、#result（結果頁，初始隱藏）。先完成 CSS 樣式再寫 JS。
渲染題目函式 renderQuestion(index)
寫一個 renderQuestion(index) 函式，用 textContent 更新題目文字，清空並重新建立選項按鈕（createElement），設定 data-index 屬性記錄選項編號，加上點擊事件。更新進度顯示（如「第 2 題，共 5 題」）。
處理選項點擊
在選項容器上使用事件委派，點擊選項時：判斷是否正確（e.target.dataset.index == questions[current].answer），用 classList 加上 .correct 或 .wrong 樣式，加分，短暫停用所有選項（避免重複點擊），1 秒後自動前進到下一題。
進度條動畫
加入進度條元素，每道題完成後更新 style.width（百分比 = 當前題數 / 總題數 × 100）。用 CSS transition: width .4s ease 讓進度條平滑增長。同時更新文字顯示「2 / 5」。
結果頁面
所有題目答完後，隱藏題目區（classList.add('hidden')），顯示結果區，用 textContent 顯示分數（如「答對 4 / 5 題」）。根據分數顯示不同的評語文字和表情。加入「重新開始」按鈕，點擊後重設所有狀態並呼叫 renderQuestion(0)。
加分功能：題目隨機排序
用 Fisher-Yates 洗牌演算法打亂題目順序：每次開始前對題目陣列呼叫 sort(() => Math.random() - 0.5)（簡易版）或完整實作洗牌。同時也對每道題的選項順序打亂，但要記得更新 answer 索引指向正確選項的新位置。
加分功能：鍵盤支援
用 document.addEventListener('keydown') 讓使用者可以按數字鍵 1–4 選擇選項，按 Enter 在結果頁重新開始。在選項按鈕上加 tabindex="0" 讓鍵盤可以 Tab 鍵導航並按 Space/Enter 選擇，提升無障礙體驗。

完成項目自評

題目資料用 JS 陣列儲存，HTML 中沒有硬編碼的題目內容
使用 textContent（非 innerHTML）更新題目文字，避免 XSS 風險
選項按鈕用 createElement 動態建立，選完後正確答案顯示綠色，錯誤顯示紅色
使用事件委派（在容器上綁定，用 e.target 判斷）而非在每個選項上個別綁定
進度條有 CSS transition 動畫，平滑更新
答完所有題目後顯示結果頁，含正確題數與評語
重新開始按鈕能正確重設所有狀態並重新渲染第一題
（加分）支援鍵盤按 1–4 選擇，Enter 重新開始
