核心概念 01

React 的思維模型

學 React 最大的挑戰不是語法，是思維方式的轉變。在開始寫程式之前，先理解 React 為什麼存在，它解決了什麼問題。

01
命令式 vs 宣告式
Imperative vs Declarative

用原生 DOM 操作（Week 05）的方式叫做命令式（Imperative）：你明確告訴瀏覽器「找到這個元素、修改它的文字、加上這個類別、移除那個元素」——你描述的是步驟。

React 採用宣告式（Declarative）：你描述「在這個狀態下，UI 應該長什麼樣子」，然後讓 React 找出需要做哪些 DOM 更新。你描述的是結果，而非過程。

這就像你可以告訴計程車司機「左轉、直走 200 公尺、右轉」（命令式），也可以直接說「帶我去台北車站」（宣告式）。React 就是那位知道最佳路線的司機。

命令式（原生 DOM）— 描述步驟
// 切換喜愛狀態，需要手動操作 DOM
let liked = false;
const btn = document.querySelector('#like-btn');

btn.addEventListener('click', () => {
  liked = !liked;
  if (liked) {
    btn.textContent = '❤️ 已喜愛';
    btn.classList.add('liked');
    btn.style.color = 'red';
  } else {
    btn.textContent = '🤍 喜愛';
    btn.classList.remove('liked');
    btn.style.color = '';
  }
  // 還要更新計數器...
  document.querySelector('#count')
    .textContent = liked ? '1 個喜愛' : '0 個喜愛';
});
宣告式（React）— 描述結果
// 描述「在此狀態下 UI 長什麼樣」
function LikeButton() {
  const [liked, setLiked] = useState(false);

  return (
    <div>
      <button onClick={() => setLiked(!liked)}
              style={{color: liked ? 'red' : 'inherit'}}>
        {liked ? '❤️ 已喜愛' : '🤍 喜愛'}
      </button>
      <span>{liked ? '1' : '0'} 個喜愛</span>
    </div>
  );
}

01
Virtual DOM 與 Reconciliation
為什麼 React 這麼快

React 在記憶體中維護一份虛擬 DOM（Virtual DOM）——一個描述 UI 的 JavaScript 物件樹。當狀態改變，React 重新計算虛擬 DOM，然後透過協調演算法（Reconciliation）比較新舊兩份虛擬 DOM 的差異（Diffing），只把真正有變化的部分更新到真實 DOM。

這樣做的好處：直接操作 DOM 很慢，但比較 JS 物件很快。React 只在必要時才動真實 DOM，避免無謂的重繪（Repaint）和重排（Reflow），讓介面更新更有效率。

React 18 的 Concurrent Mode React 18 引入了並發模式，讓 React 可以中斷、暫停、恢復渲染工作，優先處理使用者互動，讓即使在複雜計算期間介面也保持響應。這個機制在你學完基礎後值得深入探索。

核心概念 02

JSX 語法

JSX 看起來像 HTML，但它是 JavaScript 的語法擴充。每個 JSX 標籤都會被 Babel 轉譯成 React.createElement() 呼叫。理解它的規則，你就能自信地寫 JSX。

02
JSX 與 HTML 的差異
六個必須知道的規則

1. className 取代 class：因為 class 是 JS 保留字，JSX 中用 className。

2. camelCase 屬性：HTML 屬性在 JSX 中全部改為 camelCase：onclick → onClick、tabindex → tabIndex、for（label）→ htmlFor。

3. 單一根元素：每個元件只能回傳一個根元素。若不想多包一層 div，可用 Fragment：<>...</> 或 <React.Fragment>。

4. 自閉合標籤必須閉合：HTML 中 <br> 合法，但 JSX 中必須寫 <br />，所有標籤都要關閉。

5. 用 {} 插入 JS 表達式：大括號裡可以放任何 JS 表達式（變數、運算、函式呼叫），但不能放語句（if/for）。

6. style 是物件：style 接受 JS 物件而非字串：style={{color: 'red', fontSize: '16px'}}（雙大括號：外層是 JSX 表達式，內層是物件字面值）。

JSX — 語法規則完整範例
// ── JSX 被 Babel 轉譯的樣子 ──
// JSX:   <h1 className="title">Hello</h1>
// 轉譯: React.createElement('h1', {className:'title'}, 'Hello')

function MyComponent() {
  const name  = 'React';
  const count = 42;
  const isAdmin = true;
  const items = ['A', 'B', 'C'];

  return (
    {/* Fragment 避免多餘的 div */}
    <>
      {/* className 不是 class */}
      <h1 className="title">Hello, {name}!</h1>

      {/* 大括號內可放任何 JS 表達式 */}
      <p>計算結果：{count * 2}</p>
      <p>現在時間：{new Date().toLocaleTimeString()}</p>

      {/* style 是物件（雙大括號）*/}
      <div style={{
        color: isAdmin ? 'gold' : 'gray',
        fontSize: '1.2rem',         // camelCase！
        backgroundColor: '#1a1a2e'  // 不是 background-color
      }}>
        管理員模式
      </div>

      {/* htmlFor 取代 for */}
      <label htmlFor="email">信箱</label>
      <input id="email" type="email" />  {/* 自閉合要有 / */}
      <br />
      <img src="./logo.png" alt="Logo" />

      {/* 大括號不能放 if/for，用表達式替代 */}
      <span>{isAdmin ? '管理員' : '一般用戶'}</span>
      {isAdmin && <button>刪除</button>}  {/* 短路渲染 */}

      {/* 列表渲染 */}
      <ul>
        {items.map((item, i) => (
          <li key={i}>{item}</li>  // key 是必要的
        ))}
      </ul>
    </>
  );
}
常見 JSX 錯誤 JSX 中的注釋不能用 // 或 /* */ 直接放在標籤層級，需要用 {/* 注釋 */}（大括號包裹）。另外，false、null、undefined、0... 其中 0 會被渲染成文字「0」！用 count && <Comp /> 時，如果 count 是 0，畫面會顯示 0 而非不渲染。改用 count > 0 && <Comp /> 或三元運算子更安全。
核心概念 03

元件與 Props

元件是 React 的核心積木。Props 是父元件向子元件傳遞資料的唯一方式。理解這個單向資料流，是掌握 React 架構的關鍵。

03
函式元件
Function Component

現代 React 只使用函式元件（Function Component）。它就是一個普通的 JS 函式：接受 props 物件作為參數，回傳描述 UI 的 JSX。命名規則：首字母必須大寫（React 用這個判斷是元件還是 HTML 標籤）。

props（properties 縮寫）是父元件傳入的資料，在子元件內是唯讀的——絕對不能在子元件中修改 props，這是 React 最重要的規則之一（單向資料流）。如果子元件需要改變這個值，應該呼叫父元件傳入的回呼函式。

特殊 prop：children——當你在元件標籤之間放內容時，這些內容會透過 props.children 傳入，讓你可以建立像 Card、Modal、Layout 這類包裝元件。

// ── 定義元件（接受 props）──
function Card({ title, description, buttonLabel, accentColor, variant = 'default' }) {
  // 解構賦值從 props 取出各個屬性，variant 有預設值

  const styles = {
    default:  { border: `1px solid ${accentColor}` },
    featured: { border: `2px solid ${accentColor}`, boxShadow: `0 0 20px ${accentColor}40` },
    minimal:  { border: '1px solid #333' },
  };

  return (
    <div className={`card card--${variant}`} style={styles[variant]}>
      <h3 style={{color: accentColor}}>{title}</h3>
      <p>{description}</p>
      <button style={{background: accentColor}}>
        {buttonLabel}
      </button>
    </div>
  );
}

// ── 使用元件（傳入 props）──
function App() {

Props 最佳實踐 使用解構賦值取出 props（function Card({ title, desc })）比 props.title 更簡潔。給 props 設定預設值（variant = 'default'）讓元件更健壯。傳遞函式作為 props 時，慣例以 on 開頭命名（onSubmit、onChange、onDelete），清楚表明它是事件回呼。
核心概念 04

State 與 useState

State 是讓元件「有記憶」的機制——記住按鈕被點幾次、輸入框裡有什麼文字、側欄是否展開。每次 state 改變，React 重新渲染元件。

04
useState Hook
const [value, setValue] = useState(initialValue)

useState 回傳一個陣列：第一個元素是當前狀態值，第二個元素是更新函式。用解構賦值命名：const [count, setCount] = useState(0)。

呼叫 setCount(newValue) 後，React 會重新渲染這個元件（及其子元件），用新值重新計算 JSX 並更新 DOM。狀態更新不是立即的——呼叫 set 函式後，count 的值在這次渲染中仍是舊值，要到下次渲染才會是新值。

黃金規則：不能直接修改 state。對物件和陣列型別的 state，必須建立新的物件/陣列，不能用 push、splice 等方法直接修改。React 用參考比對判斷是否需要重新渲染，直接修改不會觸發重新渲染。

State 不是普通變數 普通變數的改變不會觸發重新渲染，也不會在重新渲染之間保留值。只有用 useState 管理的值，React 才會在重新渲染時記住並正確更新。如果你發現「改了值但畫面沒更新」，通常就是忘了用 state。

JSX — useState 完整參考
import { useState } from 'react';

// ── 基本使用 ──
function Counter() {
  const [count, setCount] = useState(0); // 初始值 0

  return (
    <div>
      <p>計數：{count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}

// ── 函式形式更新（依賴前一個值時必用）──
setCount(prev => prev + 1); // prev 保證是最新值

// ── 物件 state：必須展開，不能直接修改 ──
const [user, setUser] = useState({ name: 'Alice', age: 25 });

// ❌ 錯誤：直接修改，不會觸發重新渲染
user.name = 'Bob';
setUser(user); // 參考沒變，React 不會重新渲染

// ✅ 正確：建立新物件
setUser({ ...user, name: 'Bob' }); // 展開舊物件，覆寫 name

// ── 陣列 state：必須建立新陣列 ──
const [todos, setTodos] = useState([]);

// ❌ 錯誤
todos.push({ id: 1, text: '買咖啡' }); setTodos(todos);

// ✅ 新增
setTodos([...todos, { id: Date.now(), text: '買咖啡', done: false }]);

// ✅ 刪除（filter 建立新陣列）
setTodos(todos.filter(todo => todo.id !== targetId));

// ✅ 更新某一項（map 建立新陣列）
setTodos(todos.map(todo =>
  todo.id === targetId ? { ...todo, done: !todo.done } : todo
));

// ── 布林值切換 ──
const [isOpen, setIsOpen] = useState(false);
setIsOpen(prev => !prev); // 切換

// ── 字串 state（受控輸入）──
const [input, setInput] = useState('');
<input
  value={input}
  onChange={e => setInput(e.target.value)}
/>
核心概念 05

條件渲染與列表

根據狀態決定顯示什麼、把資料陣列變成 UI 元素——這兩個技巧在每個 React 應用中無處不在。

05
條件渲染的三種寫法
三元 / 短路 / 提前回傳

三元運算子（? :）：最常用，適合在 JSX 中選擇渲染兩種不同的內容。{isLoggedIn ? <Dashboard /> : <Login />}

&& 短路語法：只有 true 時才渲染，false 時不渲染任何東西（注意 0 的陷阱）。適合可選的 UI 元素：{hasError && <ErrorMessage />}

提前回傳（Early Return）：在函式最開頭判斷，直接 return null 或另一個 JSX。適合整個元件的可見性切換，讓主要渲染邏輯保持簡潔。

05
列表渲染與 key prop
Array.map() + key

在 React 中，把陣列渲染成 UI 元素的標準做法是 array.map()，回傳 JSX 陣列。每個元素必須有唯一的 key prop，讓 React 在更新時識別哪個元素是哪個，避免不必要的重新渲染或狀態錯誤。

key 應該是穩定、唯一的識別符——用資料的 ID 是最佳選擇。不應該用陣列索引（index）作為 key，因為當列表重新排序或刪除中間項目時，索引會對應到不同的資料，導致 React 的 Diffing 出錯，可能造成 UI 狀態異常。

key 不是 prop key 是 React 的特殊屬性，不會被傳入元件的 props。如果你需要在元件內使用元素的 ID，必須另外傳一個獨立的 prop（如 id={item.id}）。
JSX — 條件渲染與列表
// ── 條件渲染：三元運算子 ──
function UserPanel({ isLoggedIn, user }) {
  return (
    <div>
      {isLoggedIn ? (
        <div>
          <p>歡迎，{user.name}！</p>
          <button>登出</button>
        </div>
      ) : (
        <button>登入</button>
      )}

      {/* && 短路：只有 true 才渲染 */}
      {user.isAdmin && <AdminBadge />}

      {/* ⚠️ 避免用數字做短路（0 會渲染）*/}
      {notifications.length > 0 && <Badge count={notifications.length} />}
    </div>
  );
}

// ── 提前回傳（早期 return）──
function Profile({ user }) {
  if (!user) return null;           // 無資料時不渲染
  if (user.banned) return <BannedMessage />;
  return <div>{user.name}</div>;    // 主要渲染
}

// ── 列表渲染：用 .map() ──
const todos = [
  { id: 1, text: '學 React',   done: true  },
  { id: 2, text: '做作業',    done: false },
  { id: 3, text: '寫部落格',  done: false },
];

function TodoList({ todos }) {
  return (
    <ul>
      {todos.map(todo => (
        <li
          key={todo.id}            // ✅ 用 id 而非 index
          style={{textDecoration: todo.done ? 'line-through' : 'none'}}
        >
          {todo.text}
        </li>
      ))}
    </ul>
  );
}

// ── 過濾後渲染 ──
function FilteredList({ items, query }) {
  const filtered = items.filter(item =>
    item.name.toLowerCase().includes(query.toLowerCase())
  );

  if (filtered.length === 0) {
    return <p className="empty">找不到符合的結果</p>;
  }

  return (
    <ul>
      {filtered.map(item => (
        <li key={item.id}>{item.name}</li>
      ))}
    </ul>
  );
}
核心概念 06

React 中的事件處理

React 事件和原生 DOM 事件很相似，但有幾個關鍵差異需要理解，尤其是「傳函式引用」vs「呼叫函式」這個常見誤解。

06
SyntheticEvent 與事件處理模式
React 的事件系統

React 的事件處理器（如 onClick）使用合成事件（SyntheticEvent）——它包裝了瀏覽器原生事件，提供跨瀏覽器一致的介面。你可以用 e.preventDefault()、e.stopPropagation() 等方法，行為和原生事件相同。

最重要的規則：傳入函式引用，不要呼叫函式。onClick={handleClick} 正確（傳函式，點擊時才執行）；onClick={handleClick()} 錯誤（立刻執行，渲染時就觸發）。

當需要傳遞額外參數時，用箭頭函式包裹：onClick={() => handleDelete(item.id)}。這樣點擊時才會呼叫 handleDelete，並傳入 item.id。

JSX — 事件處理完整模式
// ── 基本事件綁定 ──
function MyForm() {
  const [value, setValue] = useState('');

  // 事件處理函式（接受 event 物件）
  function handleChange(e) {
    setValue(e.target.value);
  }

  function handleSubmit(e) {
    e.preventDefault(); // 阻止表單預設提交
    console.log('送出：', value);
  }

  return (
    <form onSubmit={handleSubmit}>  {/* 傳引用，不是呼叫 */}
      <input
        value={value}
        onChange={handleChange}    {/* ✅ 正確 */}
        {/* onChange={handleChange()} ❌ 錯誤，立刻執行 */}
      />
      <button type="submit">送出</button>
    </form>
  );
}

測試你的理解

四道題目，涵蓋本週 React 核心概念。

QUIZ 01 / 04
以下哪段程式碼「正確地」更新一個物件 state？
Auser.name = 'Bob'; setUser(user);
BsetUser(Object.assign(user, { name: 'Bob' }));
CsetUser({ ...user, name: 'Bob' });
DsetUser(prev => { prev.name = 'Bob'; return prev; });
CORRECT展開運算子 {...user, name: 'Bob'} 建立一個新物件，包含 user 的所有屬性，再覆寫 name。React 偵測到新的物件參考，觸發重新渲染。選項 A 和 B 直接修改原物件，D 在回呼函式中修改 prev——這些都沒有建立新物件，React 不會觸發重新渲染。
TRY AGAINReact 用「參考相等性」判斷 state 是否改變。只有傳入一個全新的物件，React 才會重新渲染。哪個選項建立了新物件？
QUIZ 02 / 04
渲染以下 JSX 時，畫面會顯示什麼？
{0 && <span>有訊息</span>}
A什麼都不顯示（因為 0 是 falsy）
B顯示數字 0
C顯示「有訊息」
D拋出錯誤
CORRECT這是 React 著名的陷阱！false、null、undefined 不會渲染任何東西，但數字 0 會被渲染成文字「0」。因為短路求值的結果是 0（一個數字），而 React 會把數字渲染出來。安全做法：{count > 0 && ...} 或 {!!count && ...}。
TRY AGAIN注意 React 對「falsy 值」的渲染規則——false、null、undefined 和數字 0 的處理方式不完全一樣。
QUIZ 03 / 04
在 React 列表渲染中，為什麼不建議用陣列索引（index）作為 key？
A因為 React 不支援數字類型的 key，只接受字串
B因為 index 不能保證唯一性
C因為使用 index 會讓效能變差
D因為刪除或重排列表時，index 與資料的對應關係會改變，導致 React 的 Diffing 出錯，可能造成元件狀態異常
CORRECT假設列表有 [A(0), B(1), C(2)]，刪掉 A 後變成 [B(0), C(1)]。React 看到「key=0 的內容從 A 變成 B」，會認為是更新而非刪除，可能保留 A 的元件狀態（如輸入框的值）給 B 使用。只有在列表不會重排、也不會刪除中間元素時，才可以用 index。
TRY AGAINKey 的作用是讓 React 識別「這個 DOM 節點對應的是哪筆資料」。當列表發生變化時，如果 key 的對應關係改變了，會發生什麼？
QUIZ 04 / 04
以下哪個寫法在點擊按鈕時「不會」執行 handleDelete(item.id)？
A<button onClick={() => handleDelete(item.id)}>
B<button onClick={handleDelete.bind(null, item.id)}>
C<button onClick={handleDelete(item.id)}>
Dconst del = () => handleDelete(item.id); <button onClick={del}>
CORRECT選項 C onClick={handleDelete(item.id)} 在 JSX 渲染時就立刻呼叫了 handleDelete(item.id)，把回傳值（通常是 undefined）設定為 onClick 的 handler，所以點擊時什麼都不發生。其他選項都正確地傳入一個函式，點擊時才執行。
TRY AGAIN區分「傳入函式」和「呼叫函式並傳入回傳值」。onClick 需要接受一個函式，點擊時才呼叫它。

計數器 + 待辦清單

用 Vite + React 建立真實專案，實作元件拆分、props 傳遞、useState 管理，打造一個完整的互動待辦清單應用程式。

完成步驟

用 Vite 建立 React 專案
在終端機執行 npm create vite@latest my-todo-app -- --template react，進入資料夾後執行 npm install，再 npm run dev 啟動開發伺服器。刪除 src/App.css 內容和 src/index.css 的預設樣式，從空白開始。
Part 1：實作計數器元件
建立 src/components/Counter.jsx，使用 useState(0) 管理計數。實作「加一」「減一」「重設」三個按鈕，加上讓計數值不能低於 0 的限制（Math.max(0, count - 1)）。在 App.jsx 中引入並使用此元件。
拆分元件結構
建立以下元件檔案：Header.jsx（顯示標題和待辦數量）、TodoForm.jsx（輸入框和新增按鈕）、TodoList.jsx（渲染列表）、TodoItem.jsx（單一項目）。先把 App.jsx 中的 state 和函式設計好，再決定哪些 props 需要往下傳。
在 App.jsx 實作核心邏輯
在 App.jsx 中用 useState([]) 管理 todos 陣列。實作三個函式：addTodo(text)（新增，每個 todo 需要唯一 id，可用 Date.now()）、toggleTodo(id)（切換完成狀態，用 map 建立新陣列）、deleteTodo(id)（刪除，用 filter 建立新陣列）。
實作 TodoForm（受控輸入）
在 TodoForm 中用 useState('') 管理輸入值，input 的 value 綁定 state，onChange 更新 state（受控元件模式）。送出時（按鈕 onClick 或 form onSubmit）驗證輸入非空，呼叫 props.onAdd(inputValue)，然後清空輸入框。
實作 TodoList 與 TodoItem
TodoList 接受 todos 陣列和 onToggle、onDelete 函式，用 todos.map() 渲染 TodoItem，記得傳入 key={todo.id}。TodoItem 顯示文字（完成時加刪除線樣式）、完成切換按鈕、刪除按鈕，點擊時呼叫對應的 props 函式並傳入 todo.id。
加分：篩選功能
在 App.jsx 加入 filter state（'all' | 'active' | 'done'）。渲染前根據 filter 篩選 todos：const filtered = todos.filter(t => filter === 'all' ? true : filter === 'done' ? t.done : !t.done)。建立三個篩選按鈕，active 狀態用 className 標示。
加分：LocalStorage 持久化
在初始化時從 localStorage 讀取：useState(() => JSON.parse(localStorage.getItem('todos') ?? '[]'))（lazy initializer）。每次 todos 改變時儲存，這需要 useEffect（下週主題，先嘗試了解）：useEffect(() => localStorage.setItem('todos', JSON.stringify(todos)), [todos])。
