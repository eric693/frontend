02 — Event Loop

JavaScript 為何是非同步的？

JavaScript 是單執行緒（Single-Threaded）語言，同一時間只能執行一件事。 但瀏覽器提供了 Web API（計時器、網路請求、DOM 事件），讓這些耗時操作可以在背景執行， 完成後再透過 Event Loop 把結果推回執行佇列。

EL
Event Loop 的四個角色
Call Stack / Web API / Task Queue / Event Loop

Call Stack（呼叫堆疊） 是 JS 執行程式碼的地方，函式呼叫時被推入，執行完畢後彈出。 當遇到非同步操作（如 fetch、setTimeout），該操作會被移交給 Web API 處理， Call Stack 繼續執行後續的同步程式碼，不會等待。

Web API 完成工作後，將 Callback 函式放入 Task Queue（任務佇列）。 Event Loop 持續檢查：「Call Stack 是否清空了？」——清空後才從 Task Queue 取出任務執行。 這就是非同步的核心機制。

Microtask Queue（Promise 的 callback）優先級高於 Task Queue， 每次 Call Stack 清空後，會先清空所有 Microtask，才處理 Task Queue。

CALL STACK
→
console.log("start") → 執行並彈出
CALL STACK
→
fetch(...) → 推入後移交給 Web API，立即彈出
WEB API
→
網路請求在背景執行（500ms）
CALL STACK
→
console.log("end") → 執行並彈出（不等 fetch）
MICROTASK
→
fetch 完成 → .then() callback 放入 Microtask Queue
EVENT LOOP
→
Call Stack 空了 → 取出 .then() callback 執行
JavaScript — 執行順序示範 
console.log("1 — 同步，立即執行")

setTimeout(() => {
  console.log("3 — Task Queue，最後執行")
}, 0)

Promise.resolve().then(() => {
  console.log("2 — Microtask，setTimeout 之前")
})

console.log("1.5 — 同步，立即執行")

// 輸出順序：
// 1 — 同步，立即執行
// 1.5 — 同步，立即執行
// 2 — Microtask，setTimeout 之前
// 3 — Task Queue，最後執行
⚠ 常見誤解 setTimeout(fn, 0) 並不代表「立即執行」，而是「盡快執行，但要等 Call Stack 清空並讓 Microtask 先跑完」。 在繁忙的頁面中，實際延遲可能遠超過 0ms。
03 — Callback Pattern

Callback：非同步的第一代解法

Callback 是最原始的非同步處理方式——把「完成後要做的事」作為函式傳入。 雖然直觀，但多層嵌套會形成惡名昭彰的 Callback Hell。

✗ Callback Hell（金字塔末日）
getUser(userId, (user) => {
  getPosts(user.id, (posts) => {
    getComments(posts[0].id, (comments) => {
      getLikes(comments[0].id, (likes) => {
        // 越來越深，難以維護
        console.log(likes)
      }, handleError)
    }, handleError)
  }, handleError)
}, handleError)
✓ Promise 鏈（扁平化）
getUser(userId)
  .then(user => getPosts(user.id))
  .then(posts => getComments(posts[0].id))
  .then(comments => getLikes(comments[0].id))
  .then(likes => console.log(likes))
  .catch(handleError)
// 扁平、可讀、單一 catch
CB
Callback 的正確用途
並非所有 Callback 都是問題

Callback 本身並不是反模式，問題在於多層巢狀的非同步 Callback。 對於同步操作（如 Array.forEach、Array.map、事件監聽器）， Callback 依然是標準且正確的用法。

只有當你需要「依序等待多個非同步操作」時，才需要升級到 Promise 或 async/await。

JavaScript — Node.js 風格 Callback（Error-First） 
// Node.js 慣例：第一個參數是 error，第二個是結果
function readFile(path, callback) {
  // 模擬非同步檔案讀取
  setTimeout(() => {
    if (!path) {
      callback(new Error("路徑不能為空"), null)
    } else {
      callback(null, "檔案內容...")
    }
  }, 500)
}

readFile("./data.txt", (err, data) => {
  if (err) {
    console.error("讀取失敗：", err.message)
    return
  }
  console.log("讀取成功：", data)
})
04 — Promise

Promise：非同步的優雅解法

Promise 是一個代表「未來某個時刻才會有結果」的物件。 它有三種狀態，且狀態只能轉換一次，讓非同步流程變得可預測、可組合。

⏳
PENDING
初始狀態，操作尚未完成
✅
FULFILLED
操作成功完成，有結果值
❌
REJECTED
操作失敗，有錯誤原因
JavaScript — 建立與使用 Promise 
// ── 建立 Promise ──
const fetchData = new Promise((resolve, reject) => {
  setTimeout(() => {
    const success = true
    if (success) {
      resolve({ name: "Eric", score: 98 })  // 成功 → FULFILLED
    } else {
      reject(new Error("請求失敗"))          // 失敗 → REJECTED
    }
  }, 1000)
})

// ── 消費 Promise ──
fetchData
  .then(data => {
    console.log("成功：", data)       // { name: "Eric", score: 98 }
    return data.score * 2            // 可以 return 值，傳給下個 .then
  })
  .then(doubled => console.log("Double：", doubled))  // 196
  .catch(err => console.error("失敗：", err.message))
  .finally(() => console.log("無論成敗都執行"))

Promise 的靜態方法

Promise.all()
Promise.all([p1, p2, p3])
等待全部 Promise 完成，任一 reject 就整體 reject。適合並行獨立請求。
const [user, posts] = await Promise.all([getUser(), getPosts()])
Promise.allSettled()
Promise.allSettled([p1, p2, p3])
等待全部 Promise 完成，不管成敗，每個都回傳狀態 + 值/原因。
results.forEach(r => r.status === 'fulfilled' ? ... : ...)
Promise.race()
Promise.race([p1, p2, p3])
取最快完成（或失敗）的那個，常用於設定請求逾時。
Promise.race([fetch(url), timeout(5000)])
Promise.any()
Promise.any([p1, p2, p3])
取最快成功的那個，全部 reject 才 reject，適合備援請求。
const data = await Promise.any([server1(), server2()])
JavaScript — Promise.all 並行請求 
// 並行請求三個 API（比串行快 3 倍）
const [weather, movies, news] = await Promise.all([
  fetch("https://api.example.com/weather").then(r => r.json()),
  fetch("https://api.example.com/movies").then(r => r.json()),
  fetch("https://api.example.com/news").then(r => r.json()),
])

// vs 串行（慢）：
const w = await fetch(weatherUrl).then(r => r.json()) // 等這個
const m = await fetch(moviesUrl).then(r => r.json())   // 再等這個
const n = await fetch(newsUrl).then(r => r.json())    // 再等這個
05 — async / await

async / await：讓非同步像同步一樣讀

async/await 是 ES2017 引入的語法糖，建立在 Promise 之上。 它讓非同步程式碼看起來像同步，大幅提升可讀性，是現代 JS 的標準寫法。

A
async 函式的本質
永遠回傳 Promise

在函式前加 async 關鍵字，該函式就會自動將回傳值包裝成 Promise。 即使你 return 42，外部拿到的也是 Promise.resolve(42)。

在 async 函式內部使用 await，會暫停該函式的執行，等待 Promise 解析完成， 然後取出結果值繼續執行。注意：await 只能在 async 函式內使用（模組頂層除外）。

Promise 鏈寫法
function getUserData(id) {
  return fetchUser(id)
    .then(user => fetchPosts(user.id))
    .then(posts => {
      return { posts, count: posts.length }
    })
    .catch(err => {
      console.error(err)
      throw err
    })
}
✓ async/await 寫法
async function getUserData(id) {
  try {
    const user = await fetchUser(id)
    const posts = await fetchPosts(user.id)
    return { posts, count: posts.length }
  } catch (err) {
    console.error(err)
    throw err
  }
}
JavaScript — 常見 async/await 技巧 
// ── 1. 頂層 await（ES Module）──
const data = await fetch(url).then(r => r.json())

// ── 2. 並行 await（正確方式）──
const [a, b] = await Promise.all([taskA(), taskB()])

// ── 3. 迴圈中的 await（注意！）──
const ids = [1, 2, 3]

// ✗ forEach 不支援 async callback，await 無效
ids.forEach(async id => {
  await processId(id) // 不會依序等待！
})

// ✓ 串行：用 for...of
for (const id of ids) {
  await processId(id) // 依序執行 ✓
}

// ✓ 並行：用 Promise.all + map
await Promise.all(ids.map(id => processId(id)))
💡 最佳實踐 不要寫 await somePromise().then(...)——混用 await 和 .then() 讓程式碼風格不一致。 選擇一種並堅持使用：現代專案中推薦全面使用 async/await + try/catch。
06 — Fetch API

Fetch API：瀏覽器內建的 HTTP 客戶端

fetch() 是瀏覽器提供的原生 API，用來發送 HTTP 請求。 它取代了舊式的 XMLHttpRequest，以 Promise 為基礎，語法簡潔直觀。

FS
fetch() 的重要特性
只有網路錯誤才 reject，HTTP 4xx / 5xx 不會

fetch() 只有在真正的網路錯誤（如無網路連線、DNS 失敗）時才會 reject。 即使伺服器回傳 404 或 500，fetch 依然是 fulfilled 的狀態。

因此必須手動檢查 response.ok（status 在 200–299 時為 true）， 或檢查 response.status 的值，才能正確處理 HTTP 錯誤。

JavaScript — GET 請求（完整版） 
async function getWeather(city) {
  const API_KEY = "your_api_key"
  const url = `https://api.openweathermap.org/data/2.5/weather?q=${city}&appid=${API_KEY}&units=metric&lang=zh_tw`

  try {
    const response = await fetch(url)

    // ⚠ 必須手動檢查 HTTP 狀態碼
    if (!response.ok) {
      throw new Error(`HTTP 錯誤：${response.status} ${response.statusText}`)
    }

    const data = await response.json()  // 解析 JSON Body
    return data
  } catch (err) {
    if (err.name === "TypeError") {
      throw new Error("網路連線失敗，請檢查網路")
    }
    throw err  // 重新拋出，讓呼叫者處理
  }
}
JavaScript — POST 請求（送資料給伺服器） 
async function createPost(postData) {
  const response = await fetch("https://jsonplaceholder.typicode.com/posts", {
    method: "POST",              // HTTP 方法
    headers: {
      "Content-Type": "application/json",  // 告訴伺服器 Body 格式
      "Authorization": `Bearer ${token}`    // JWT Token
    },
    body: JSON.stringify(postData)  // 物件轉成 JSON 字串
  })

  if (!response.ok) {
    const error = await response.json()
    throw new Error(error.message || "新增失敗")
  }

  return await response.json()
}

// 使用範例
const newPost = await createPost({
  title: "我的第一篇文章",
  body: "內容...",
  userId: 1
})

Response 解析方法

方法 用途 回傳型別
response.json() 解析 JSON 格式的 Body（最常用） Promise<any>
response.text() 讀取純文字 Body（HTML、CSV） Promise<string>
response.blob() 讀取二進位資料（圖片、檔案） Promise<Blob>

⚠ 注意 Response Body 只能被讀取一次。呼叫 response.json() 後， 再呼叫 response.text() 會拋出錯誤。如需多次讀取，先呼叫 response.clone()。
07 — Error Handling

健壯的錯誤處理策略

良好的錯誤處理讓使用者看到有意義的提示，而非白屏或靜默失敗。 以下是在串接 API 時的完整錯誤處理架構。

JavaScript — 分層錯誤處理架構 
// ── 第一層：API 封裝層，統一錯誤格式 ──
class ApiError extends Error {
  constructor(message, status, data) {
    super(message)
    this.name = "ApiError"
    this.status = status
    this.data = data
  }
}

async function apiFetch(url, options = {}) {
  const response = await fetch(url, options)
  if (!response.ok) {
    let errorData
    try { errorData = await response.json() } catch {}
    throw new ApiError(
      errorData?.message || `HTTP ${response.status}`,
      response.status,
      errorData
    )
  }
  return response.json()
}

// ── 第二層：UI 層，依錯誤類型給不同提示 ──
async function loadWeather(city) {
  showLoading()
  try {
    const data = await apiFetch(`/api/weather?city=${city}`)
    renderWeather(data)
  } catch (err) {
    if (err instanceof ApiError) {
      switch (err.status) {
        case 404: showError(`找不到城市「${city}」，請確認拼字`); break
        case 401: showError("API 金鑰無效"); break
        case 429: showError("請求次數超限，請稍後再試"); break
        default:  showError(`伺服器錯誤 (${err.status})`)
      }
    } else {
      showError("網路連線失敗，請檢查網路狀態")
    }
  } finally {
    hideLoading()
  }
}
✅ 最佳實踐 永遠在 finally 中隱藏 Loading 狀態，確保無論成功或失敗，UI 都能恢復正常。 善用自訂 Error class 讓錯誤類型可被 instanceof 識別，避免靠字串比對判斷錯誤。

08 — Quiz

隨堂測驗

測試你對非同步概念的掌握程度。點選你認為正確的答案。

QUESTION 01 / 04
以下程式碼的輸出順序為何？
console.log('A'); setTimeout(() => console.log('B'), 0); Promise.resolve().then(() => console.log('C')); console.log('D');
AA → B → C → D
BA → D → C → B
CA → D → B → C
DA → C → D → B
正確！同步程式碼（A、D）先執行，接著清空 Microtask Queue（Promise .then → C），最後才執行 Task Queue（setTimeout → B）。
QUESTION 02 / 04
fetch() 發出請求後，伺服器回傳 404 狀態碼，此時 Promise 的狀態為？
Arejected（因為 404 是錯誤）
Bfulfilled（fetch 只有網路層錯誤才 reject）
Cpending（等待重試）
D取決於伺服器的回應格式
正確！fetch() 只有在無法發出請求（網路斷線、DNS 失敗等）時才 reject。HTTP 錯誤狀態碼（4xx、5xx）仍然會 fulfill，需手動檢查 response.ok。
QUESTION 03 / 04
以下哪種方式能正確地並行發出多個 fetch 請求，且等待全部完成後再處理結果？
A用 for...of 加 await 逐一發送
B用 forEach 加 async callback
C用 Promise.all() 搭配 map 建立 Promise 陣列
D連續呼叫 fetch，最後 await 最後一個
正確！Promise.all(urls.map(url => fetch(url))) 會同時發出所有請求，並等待全部完成。for...of + await 是串行的（逐一等待），效率較差。
QUESTION 04 / 04
關於 async/await，下列哪個敘述不正確？
Aasync 函式永遠回傳一個 Promise
Bawait 只能在 async 函式內使用（ES Module 頂層除外）
Casync/await 是全新的非同步機制，與 Promise 無關
Dawait 後面可以接任何 Promise 或 thenable 物件
正確！async/await 是 Promise 的語法糖，底層完全基於 Promise。它不是新的機制，只是讓 Promise 的使用更加簡潔易讀。
