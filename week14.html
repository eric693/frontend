核心概念 01

Next.js App Router 結構

Next.js 13+ 引入的 App Router 讓前後端可以在同一個專案中共存。理解它的資料夾結構，是建立全端應用的第一步。

01
App Router 資料夾慣例
app/ 目錄結構完整解說

Next.js App Router 以資料夾結構定義路由，每個資料夾代表一個路由段（route segment）。資料夾裡的特定檔名有特殊功能：page.tsx 是頁面、layout.tsx 是共用版面、route.ts 是 API 端點、loading.tsx 是載入狀態、error.tsx 是錯誤處理。

Server Components（預設）：在伺服器端渲染，可以直接存取資料庫、讀取環境變數，不能使用 useState、useEffect 等 React Hook。Client Components：在瀏覽器端渲染，需要在檔案頂端加 'use client'，可以使用 Hook 和事件處理。

重要原則：盡量讓元件在伺服器端渲染（更快、更安全），只有需要互動的部分才加 'use client'。這和 React SPA 的思維完全相反。

App Router vs Pages Router Next.js 有兩套路由系統：舊的 Pages Router（pages/ 目錄）和新的 App Router（app/ 目錄）。新專案應該使用 App Router，它支援 React Server Components，效能更好。網路上很多舊教學用 Pages Router，注意區別。

核心概念 02

Next.js API Route Handlers

API Route 讓你在 Next.js 專案中直接撰寫後端邏輯，不需要另外建立 Express 伺服器。它們在伺服器端執行，可以安全地存取資料庫和環境變數。

02
Route Handler 語法
app/api/<路徑>/route.ts

在 app/api/ 目錄建立 route.ts 檔案，匯出以 HTTP 方法命名的函式（GET、POST、PUT、DELETE）。每個函式接受一個 Request 物件，回傳一個 Response。

Next.js 的 NextResponse 提供便利的靜態方法：NextResponse.json(data, { status: 200 }) 自動設定 Content-Type 並序列化資料。錯誤回應用 NextResponse.json({ error: '...' }, { status: 400 })。

動態路由：在資料夾名稱加方括號 [id]，在 route handler 的第二個參數 params 中取得：params.id。

TypeScript — Route Handler 完整範例
// app/api/posts/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { createServerClient } from '@/lib/supabase/server';

// GET /api/posts — 取得所有文章
export async function GET(request: NextRequest) {
  try {
    const supabase = createServerClient();
    const { searchParams } = new URL(request.url);
    const limit = searchParams.get('limit') ?? '10';

    const { data, error } = await supabase
      .from('posts')
      .select('id, title, created_at, users(name)')
      .order('created_at', { ascending: false })
      .limit(Number(limit));

    if (error) throw error;

    return NextResponse.json({ data });
  } catch (err) {
    return NextResponse.json(
      { error: '無法取得文章' },
      { status: 500 }
    );
  }
}

// POST /api/posts — 新增文章
export async function POST(request: NextRequest) {
  try {
    const body = await request.json();
    const { title, content } = body;

    // 基本驗證
    if (!title?.trim()) {
      return NextResponse.json(
        { error: '標題不能為空' },
        { status: 400 }
      );
    }

    const supabase = createServerClient();
    const { data, error } = await supabase
      .from('posts')
      .insert({ title, content })
      .select()
      .single(); // 回傳新增的那一筆

    if (error) throw error;

    return NextResponse.json({ data }, { status: 201 });
  } catch (err) {
    return NextResponse.json(
      { error: '新增失敗' },
      { status: 500 }
    );
  }
}

// app/api/posts/[id]/route.ts — 動態路由
export async function DELETE(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const { id } = params; // 從 URL /api/posts/42 取得 "42"
  const supabase = createServerClient();
  const { error } = await supabase
    .from('posts')
    .delete()
    .eq('id', id);

  if (error) return NextResponse.json({ error }, { status: 500 });
  return NextResponse.json({ success: true });
}
HTTP 方法慣例用途回應狀態碼
GET取得資源（單筆或列表）200 OK
POST建立新資源201 Created
PUT完整更新資源（全部欄位）200 OK
PATCH部分更新資源（只改動的欄位）200 OK
DELETE刪除資源200 OK 或 204 No Content
任何方法驗證失敗（缺少必填欄位）400 Bad Request
任何方法未登入401 Unauthorized
任何方法沒有權限403 Forbidden
任何方法找不到資源404 Not Found
任何方法伺服器內部錯誤500 Internal Server Error
核心概念 03

Supabase 設定

Supabase 是 Firebase 的開源替代品，提供 PostgreSQL 資料庫、身份驗證、即時訂閱和儲存服務。它有友善的 UI 和 JavaScript SDK，非常適合全端入門。

03
Supabase Client 初始化
瀏覽器端 vs 伺服器端

Supabase 有兩個重要的密鑰：anon key（公開，可在前端使用，受 RLS 保護）和 service_role key（私密，只能在伺服器端使用，可繞過 RLS）。

永遠不要在前端程式碼中使用 service_role key。它有完整的資料庫存取權限，如果外洩，任何人都能讀取或刪除你所有的資料。

Next.js 中需要兩個不同的 Supabase client：瀏覽器端（createBrowserClient）用在 Client Component，使用 anon key；伺服器端（createServerClient）用在 API Route 和 Server Component，可以使用 service_role key 進行管理操作。

TypeScript — Supabase Client 設定
// ── 1. 安裝 ──
// npm install @supabase/supabase-js @supabase/ssr

// ── 2. .env.local ──
NEXT_PUBLIC_SUPABASE_URL=https://xyzxyz.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGci...  # 公開，前端可用
SUPABASE_SERVICE_ROLE_KEY=eyJhbGci...       # 私密，只能後端用！

// ── 3. lib/supabase/client.ts（瀏覽器端）──
import { createBrowserClient } from '@supabase/ssr';

export function createClient() {
  return createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  );
}

// ── 4. lib/supabase/server.ts（伺服器端）──
import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

export function createClient() {
  const cookieStore = cookies();
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        get(name) { return cookieStore.get(name)?.value; },
        set(name, value, options) { cookieStore.set(name, value, options); },
        remove(name, options) { cookieStore.set(name, '', options); },
      },
    }
  );
}

// ── 5. 在 API Route 中使用 Admin Client ──
import { createClient as createAdmin } from '@supabase/supabase-js';

function getAdminClient() {
  return createAdmin(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!  // 私密！只在伺服器用
  );
}
核心概念 04

資料庫設計

好的資料庫設計是應用穩定運行的基礎。理解資料表結構、欄位型別和關聯，是從前端思維切換到全端思維的重要一步。

04
常用 PostgreSQL 欄位型別
Supabase / PostgreSQL 型別速查

uuid：通用唯一識別符，Supabase 預設用 gen_random_uuid() 自動生成，適合所有主鍵（比遞增數字 ID 更安全，不會暴露資料量）。

text：無長度限制的字串，PostgreSQL 中 text 和 varchar 效能相同，建議統一用 text 而非 varchar。int8 / bigint：大整數，適合計數器。numeric：精確小數，適合金額。

timestamptz：帶時區的時間戳記，永遠用 timestamptz 而非 timestamp，Supabase 建議用 now() 作為預設值。bool：布林值。jsonb：可查詢的 JSON 物件，適合儲存彈性結構的資料。

-- ── 建立 todos 資料表 ──
CREATE TABLE todos (
  id         uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    uuid        NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title      text        NOT NULL,
  is_done    boolean     NOT NULL DEFAULT false,
  priority   smallint    NOT NULL DEFAULT 0,
  due_date   timestamptz,                    -- nullable，可選
  created_at timestamptz NOT NULL DEFAULT now()
);

-- ── 建立索引（加快查詢速度）──
CREATE INDEX todos_user_id_idx ON todos(user_id);
CREATE INDEX todos_created_at_idx ON todos(created_at DESC);

-- ── 常用 Supabase Dashboard SQL 查詢 ──

-- 查看所有資料
SELECT * FROM todos ORDER BY created_at DESC LIMIT 20;

-- 加入關聯查詢（JOIN）
SELECT
  t.id, t.title, t.is_done,
  u.email AS user_email
FROM todos t
JOIN auth.users u ON t.user_id = u.id
WHERE t.is_done = false
ORDER BY t.priority DESC;
核心概念 05

Supabase CRUD 操作

Supabase JavaScript SDK 提供鏈式 API，讓資料庫操作和寫原生 SQL 一樣強大，但語法更像 JavaScript。掌握這套 API 就能處理所有資料操作需求。

05
永遠處理 error
Supabase 回傳 { data, error } 解構模式

Supabase SDK 的每個操作都回傳 { data, error } 物件。操作失敗時 data 是 null，error 有值；操作成功時 error 是 null，data 有值。永遠要先檢查 error，再使用 data。

常見的錯誤處理模式：用 if (error) throw error; 或 if (error) return NextResponse.json({ error }, { status: 500 }); 中止執行，不要假設操作一定成功。

核心概念 06

Row Level Security（RLS）

RLS 是 PostgreSQL 的安全機制，讓你在資料庫層面控制「哪個使用者可以存取哪些資料列」。即使前端程式碼有漏洞，RLS 也能保護你的資料不被未授權存取。

06
為什麼 RLS 很重要
在資料庫層面保護資料

沒有 RLS 時，如果你的前端程式碼有一個 bug（例如忘記在查詢中加入 user_id 條件），使用者可能會看到所有人的資料。RLS 是最後一道防線——即使應用層的程式碼出錯，資料庫也不會回傳不屬於當前使用者的資料。

RLS Policy 的核心是一個 SQL 條件式，決定哪些資料列對當前使用者可見。Supabase 提供 auth.uid() 函式取得當前登入使用者的 ID，讓你可以輕鬆建立「只能看自己資料」的 Policy。

RLS 的預設行為 啟用 RLS 後，如果沒有任何 Policy，所有使用者對所有資料的操作都會被拒絕。你必須明確建立允許操作的 Policy。Supabase Dashboard 中每個資料表都可以看到 RLS 狀態和 Policy 列表。

SQL — RLS Policy 設定完整範例
-- ── 1. 啟用 RLS（必須先做）──
ALTER TABLE todos ENABLE ROW LEVEL SECURITY;

-- ── 2. 建立 SELECT Policy（只能查自己的資料）──
CREATE POLICY "Users can view own todos"
  ON todos
  FOR SELECT
  USING (auth.uid() = user_id);
-- auth.uid() 回傳當前登入使用者的 ID
-- USING 條件決定哪些資料列對當前使用者可見

-- ── 3. INSERT Policy（只能新增自己的資料）──
CREATE POLICY "Users can insert own todos"
  ON todos
  FOR INSERT
  WITH CHECK (auth.uid() = user_id);
-- WITH CHECK 在寫入時驗證（INSERT/UPDATE 用）

-- ── 4. UPDATE Policy（只能更新自己的資料）──
CREATE POLICY "Users can update own todos"
  ON todos
  FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- ── 5. DELETE Policy（只能刪除自己的資料）──
CREATE POLICY "Users can delete own todos"
  ON todos
  FOR DELETE
  USING (auth.uid() = user_id);

-- ── 進階：公開資料表（任何人可讀）──
CREATE POLICY "Public read access"
  ON posts
  FOR SELECT
  USING (true);  -- 條件永遠為 true = 所有人可讀

-- ── 查看目前的 Policy ──
SELECT * FROM pg_policies WHERE tablename = 'todos';

Dsrc/routes/todos.ts（Express 風格）
CORRECTNext.js App Router 的 API Route 需要放在 app/api/ 目錄下，檔名固定為 route.ts（或 .js），匯出以 HTTP 方法命名的函式（GET、POST 等）。選項 A 是舊版 Pages Router 的寫法，在 App Router 中不適用。
TRY AGAIN注意 App Router 的特殊命名慣例：特定功能的檔案有固定名稱（page.tsx、route.ts、layout.tsx），且必須放在 app/ 目錄下。
QUIZ 02 / 04
執行 Supabase 查詢後，以下哪個錯誤處理方式是正確的？
Aconst data = await supabase.from('todos').select('*'); console.log(data);
Bconst { data } = await supabase.from('todos').select('*'); if (!data) throw new Error();
Cconst { data, error } = await supabase.from('todos').select('*'); if (error) throw error;
Dtry { const data = await supabase.from('todos').select('*'); } catch(e) { console.error(e); }
CORRECTSupabase SDK 永遠回傳 {'{'} data, error {'}'} 物件，不會 throw exception（除非網路完全失敗）。正確做法是解構取出 data 和 error，先檢查 error。選項 B 不夠安全——data 在查詢成功但結果為空時是空陣列（truthy），不是 null；選項 D 無法捕捉 Supabase 層級的錯誤。
TRY AGAINSupabase SDK 的設計哲學：不拋出 exception，而是在回傳物件中包含 error。你需要解構 { data, error } 並主動檢查。
QUIZ 03 / 04
你在 Supabase 啟用了 todos 資料表的 RLS，但還沒建立任何 Policy。此時一個已登入的使用者呼叫 supabase.from('todos').select('*')，結果是？
A回傳所有 todos 資料（RLS 不影響已登入使用者）
B回傳空陣列（RLS 啟用但無 Policy = 全部拒絕）
C回傳 403 Forbidden 錯誤
D回傳只屬於當前使用者的 todos
CORRECTRLS 啟用後，預設行為是「拒絕所有操作」——除非有明確的 Policy 允許。沒有 Policy 的 RLS 就像一道完全關閉的大門，即使你已登入也無法進入。這也是為什麼啟用 RLS 後，要立刻為每種操作（SELECT/INSERT/UPDATE/DELETE）建立對應的 Policy。
TRY AGAIN想想 RLS 的「預設拒絕」原則——沒有明確允許就是拒絕，這是安全系統的基本設計哲學。
QUIZ 04 / 04
SUPABASE_SERVICE_ROLE_KEY 和 NEXT_PUBLIC_SUPABASE_ANON_KEY 的主要差異是？
A兩者功能相同，只是命名不同
BANON_KEY 只能讀資料，SERVICE_ROLE_KEY 才能寫入資料
CSERVICE_ROLE_KEY 速度更快，適合生產環境
DANON_KEY 受 RLS 保護（適合前端），SERVICE_ROLE_KEY 可繞過 RLS（只能在伺服器端使用）
CORRECT最關鍵的區別：ANON_KEY 代表「匿名/登入使用者」，受 RLS Policy 保護，可以安全地在前端使用（NEXT_PUBLIC_ 前綴讓它被打包進瀏覽器程式碼）。SERVICE_ROLE_KEY 代表「管理員」，完全繞過 RLS，絕對不能在前端程式碼中使用——如果洩露，任何人都能存取所有資料。
TRY AGAIN注意環境變數名稱的差異：有 NEXT_PUBLIC_ 前綴的會被打包進前端；沒有的只在伺服器端可見。這個命名本身就是一個安全提示。

完成步驟

建立 Next.js 專案
執行 npx create-next-app@latest my-side-project，選擇：TypeScript → Yes、ESLint → Yes、Tailwind CSS → Yes、src/ directory → No、App Router → Yes、import alias → Yes（保持 @/*）。進入目錄執行 npm run dev 確認專案正常啟動。
建立 Supabase 專案並取得金鑰
前往 supabase.com 建立免費帳號，點擊「New Project」，選擇距離最近的資料庫區域（推薦 Southeast Asia）。等待專案建立後，前往 Settings → API，複製 Project URL、anon public key 和 service_role key（小心保管！）。
安裝 Supabase SDK 並設定環境變數
執行 npm install @supabase/supabase-js @supabase/ssr。建立 .env.local，填入三個環境變數（URL、anon key、service_role key）。建立 lib/supabase/client.ts 和 lib/supabase/server.ts，參考概念 03 的程式碼。
在 Supabase 建立資料表
前往 Supabase Dashboard → SQL Editor，根據你的 Side Project 設計的 Schema（Week 13 的 Wireframe 成果），執行 CREATE TABLE SQL。建立後在 Table Editor 確認資料表結構正確，嘗試手動插入幾筆測試資料。
啟用 RLS 並建立 Policy
在 Supabase Dashboard → Authentication → Policies 中，為你的主要資料表啟用 RLS，建立四條 Policy（SELECT/INSERT/UPDATE/DELETE），全部使用 auth.uid() = user_id 條件（若你的資料表有 user_id 欄位）。確認在 Supabase Table Editor 用普通使用者查詢時無法看到其他使用者的資料。
建立 CRUD API Routes
建立 app/api/[資料表名稱]/route.ts，實作 GET（取得列表）和 POST（新增）。建立 app/api/[資料表名稱]/[id]/route.ts，實作 GET（取得單筆）、PUT（更新）、DELETE（刪除）。每個函式都要有 try-catch 和適當的 HTTP 狀態碼。
從前端串接 API
在一個 Client Component 中，用 useEffect 呼叫 fetch('/api/[路徑]') 取得資料並顯示。實作一個簡單的新增表單，送出時呼叫 POST API。使用 useState 管理 loading、error、data 三個狀態，確保使用者體驗完整。
加分：用 Supabase TypeScript 型別生成
安裝 Supabase CLI：npm install -g supabase，登入後執行 supabase gen types typescript --project-id 你的專案ID 生成 TypeScript 型別定義，儲存到 types/database.ts。在 Supabase Client 中傳入型別參數：createClient<Database>()，所有查詢都會有完整的型別檢查。
