核心概念 01

最終部署前檢查

上線前的最後一關。把這些每一項都確認過，確保你的作品在所有使用者眼中都是完整的。

01
QA 測試流程
系統性地找出所有問題

功能測試：逐一測試每個功能——登入、登出、新增、編輯、刪除、搜尋。用不同身份（新用戶、有資料的用戶、admin）測試。特別測試邊緣情況：空白輸入、超長文字、網路斷線、快速連點按鈕。

跨裝置測試：用 Chrome DevTools 的裝置模擬測試 iPhone、Samsung、iPad 三種尺寸。有條件的話，在真實手機上測試——模擬器和真機體驗有差異，尤其是觸控精確度和捲動行為。

跨瀏覽器測試：至少在 Chrome 和 Safari 兩個瀏覽器中測試。Safari 對 CSS 的支援有些差異，特別是 flex gap、backdrop-filter 等屬性。如果條件允許，在 Firefox 也測試一遍。

最容易忽略的測試場景 登入狀態過期後的行為（Token 失效時畫面不能白屏）、多個 tab 同時開啟的一致性、快速連點送出按鈕（確認有 debounce 或 disabled 防止重複請求）、貼上超長文字到輸入框（確認有長度驗證）。

核心概念 02

效能優化基礎

效能直接影響使用者體驗和搜尋排名。不需要追求完美分數，但要理解 Lighthouse 的各項指標，並知道最有效的優化方向。

02
Core Web Vitals
Google 的三個核心效能指標

LCP（Largest Contentful Paint）：最大內容渲染時間，衡量「使用者什麼時候看到頁面主要內容」。目標 < 2.5 秒。影響因素：圖片大小、伺服器回應速度、字體載入。最有效的優化：壓縮圖片（使用 WebP 格式）、設定 priority prop 給首屏最大的圖片。

FID/INP（Interaction to Next Paint）：使用者互動到畫面回應的時間。目標 < 200ms。影響因素：主執行緒被阻塞（長時間的 JS 運算）。

CLS（Cumulative Layout Shift）：累積版面位移，衡量「頁面載入過程中內容是否亂跳」。目標 < 0.1。最常見原因：圖片沒有設定寬高（瀏覽器不知道要留多少空間）、字體載入後文字重排。優化：所有 <img> 都要加 width 和 height 屬性。

Next.js 的效能優勢 Next.js 內建的 Image 元件自動處理：格式轉換（WebP/AVIF）、尺寸調整、lazy loading、避免 CLS（自動保留空間）。把所有 <img> 換成 <Image />（從 next/image 引入），是最容易獲得效能分數的優化。
Next.js — 效能優化實踐清單
// ── 1. Image 元件（取代 <img>）──
import Image from 'next/image';

// ❌ 原生 img（沒有優化）
<img src="/hero.jpg" alt="Hero" />

// ✅ Next.js Image（自動優化）
<Image
  src="/hero.jpg"
  alt="Hero"
  width={1200}
  height={600}
  priority      {/* 首屏最大圖片加這個，預先載入 */}
  quality={85}  {/* 預設 75，可依需求調整 */}
/>

// ── 2. 動態載入（Code Splitting）──

// 延遲載入不在首屏的重型元件
const HeavyChart = dynamic(() => import('./HeavyChart'), {

const inter = Inter({
  subsets: ['latin'],
  display: 'swap',  // 在字體載入前用系統字體，不閃白

核心概念 03

SEO 基礎設定

讓你的作品被搜尋引擎找到，並在社群媒體分享時顯示漂亮的預覽卡片——這些設定只需要一次，但效果長久。

03
Next.js Metadata API
統一管理 SEO 和 Open Graph

Meta Tags：title 和 description 是最基本的 SEO 設定，影響搜尋結果的標題和摘要。Description 建議 150–160 個字元，包含關鍵字但要自然。

Open Graph：當你的連結被分享到 Facebook、LINE、Discord 時，顯示的預覽卡片由 Open Graph meta tags 決定。og:image 建議尺寸 1200×630px，讓分享看起來專業。

Sitemap 和 robots.txt：sitemap.xml 告訴搜尋引擎你的網站有哪些頁面；robots.txt 告訴爬蟲哪些頁面可以索引。Next.js App Router 支援用程式碼動態生成這兩個檔案。

Next.js — 完整 SEO 設定
// ── app/layout.tsx：全站預設 Metadata ──
export const metadata: Metadata = {
  metadataBase: new URL('https://your-app.vercel.app'),

  title: {
    default:  'MyTodo App',
    template: '%s — MyTodo App',
  },
  description: '一個使用 Next.js 14、React、Supabase 建立的全端待辦事項應用',

  // Open Graph（FB / LINE / Discord 分享卡片）
  openGraph: {
    type:        'website',
    locale:      'zh_TW',
    siteName:    'MyTodo App',
    images: [{
      url:    '/og-image.jpg',   // public/og-image.jpg 1200×630
      width:  1200,
      height: 630,
      alt:    'MyTodo App 預覽圖',
    }],
  },

  // Twitter Card
  twitter: {
    card:  'summary_large_image',
    title: 'MyTodo App',
    description: '全端待辦事項應用',
  },

  // 告訴搜尋引擎網站的規範 URL
  alternates: { canonical: 'https://your-app.vercel.app' },
};

// ── 子頁面動態 Metadata ──
// app/posts/[id]/page.tsx
export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const post = await getPost(params.id);
  if (!post) return { title: '文章不存在' };
  return {
    title:       post.title,
    description: post.content.slice(0, 150),
    openGraph:   { title: post.title, images: [post.coverImage] },
  };
}

// ── app/sitemap.ts ──
export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const posts = await getAllPosts();
  const postUrls = posts.map(post => ({
    url:          `https://your-app.vercel.app/posts/${post.id}`,
    lastModified: post.updatedAt,
    changeFrequency: 'weekly' as const,
    priority:     0.8,
  }));
  return [
    { url: 'https://your-app.vercel.app', changeFrequency: 'daily', priority: 1 },
    ...postUrls,
  ];
}

// ── app/robots.ts ──
export default function robots(): MetadataRoute.Robots {
  return {
    rules: [{ userAgent: '*', allow: '/' }],
    sitemap: 'https://your-app.vercel.app/sitemap.xml',
  };
}

FINAL QUIZ 01 / 04
你的 Lighthouse Performance 分數是 55 分。以下哪個優化措施通常最能快速提升 LCP（最大內容渲染時間）？
A把所有 CSS 改為 inline style，減少外部檔案請求
B把首屏最大的 <img> 換成 Next.js <Image priority />，並確認圖片格式是 WebP
C把所有 console.log 從程式碼中移除
D增加更多 useCallback 和 useMemo 減少重新渲染
CORRECTLCP 最大的影響因素通常是首屏最大的圖片（Hero 圖、封面圖）的載入速度。priority prop 讓瀏覽器預先載入這張圖片，而不是等到渲染時才開始下載；WebP 格式讓圖片大小減少 30–50%。這通常是最容易、效果最大的 LCP 優化。
TRY AGAINLCP 測量的是最大內容元素的渲染時間，通常是圖片。想想什麼會讓圖片載入更快？

面試官看到你的 GitHub 連結後，他最可能在 30 秒內做什麼決定？哪種 README 結構最能讓他繼續看下去？
A先讀完所有技術文件，再決定要不要點 Demo

FINAL QUIZ 03 / 04
你的 Next.js 應用有一個文章列表頁，文章內容每小時更新一次。以下哪種 fetch 快取策略最合適？
Acache: 'no-store'——每次請求都取最新資料
Bnext: {'{'} revalidate: 3600 {'}'}——每小時重新驗證一次快取
C不設定任何快取選項，用 Next.js 預設
Dnext: {'{'} revalidate: 0 {'}'}——立即使快取失效
CORRECTrevalidate: 3600（3600 秒 = 1 小時）是最適合這個情境的策略：第一個訪問者觸發伺服器渲染並快取結果；之後 1 小時內的所有訪客都得到快取版本（超快！）；1 小時後快取過期，下一個請求重新取資料並更新快取。這是 ISR（Incremental Static Regeneration）模式，兼顧效能和資料新鮮度。
TRY AGAIN思考「快取時間應該和資料更新頻率匹配」這個原則——如果資料每小時更新，快取應該多久過期一次？

這是你的畢業任務。完成這些步驟，你的 Side Project 就正式上線，你正式成為一個有作品的前端開發者。

GRADUATION MISSION
Side Project
完整交付
從功能完整到完美交付——最後的部署、效能優化、SEO、作品集整理，讓全世界都能看到你的作品。

任務步驟

最終 QA 測試
逐功能測試，特別測試邊緣情況：空白輸入、網路斷線、未登入狀態直接訪問受保護頁面、Token 失效後的處理。在 Chrome、Safari 和手機上各走一遍主要功能流程。記下所有 bug 並修復。
Lighthouse 審查與優化
用 Chrome DevTools 的 Lighthouse 分析主要頁面（首頁、資料列表頁）。把所有 <img> 換成 <Image />。修正 Accessibility 警告（alt text、color contrast、keyboard navigation）。目標 Performance > 75、Accessibility > 90。
設定 SEO Metadata
在 app/layout.tsx 設定 title template、description、Open Graph image（1200×630px 的 og-image.jpg）。為主要動態頁面實作 generateMetadata。建立 app/sitemap.ts 和 app/robots.ts。用 opengraph.xyz 確認分享預覽正確。
完善 GitHub README
在 README 第一行放 Live Demo badge。加入 3 張截圖（桌面版、手機版、主要功能的 GIF）。完整的技術 Stack 表格、功能列表、安裝步驟。加一段「技術挑戰與解法」，說明你在開發中遇到的有趣問題和解決思路。
最終部署與驗收
Push 最後的程式碼到 GitHub main 分支，等待 Vercel 自動部署完成。在 Vercel 控制台確認 Deployment 成功（無 Build Error）。用無痕視窗訪問你的 Vercel URL，確認未登入狀態、登入、所有功能均正常。把連結分享給朋友、家人測試。
更新個人資料
更新 LinkedIn 的技術技能（HTML、CSS、JavaScript、React、Next.js、Supabase、Git）、工作經歷或專案欄位。把 GitHub 個人頁面整理好（個人 README、pin 最好的 3 個倉庫）。如果有個人網站，加入這個 Side Project 的展示。
寫一篇回顧文章（選做）
在 Medium、Dev.to 或個人部落格寫一篇「18 週前端學習回顧」——你學了什麼、遇到什麼困難、如何克服。這既是對自己的總結，也是很好的曝光。讀者喜歡看到真實的學習歷程，而不只是成功結果。
開始下一個旅程
根據你的目標選擇方向：找工作——整理履歷、開始投遞初階職缺、準備技術面試；繼續深化——選一個新技術（TypeScript、Testing、Docker）做下一個專案；自由接案——在 Freelancer、Upwork 或人力銀行上架你的服務。每條路都是有效的，選一條，走下去。
