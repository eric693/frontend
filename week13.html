02 — Topic Selection

選題策略

好的 Side Project 題目需要平衡三件事：你的技術能力範圍、 能在六週內完成，以及有實際的使用價值。 以下是評估一個題目的三個維度。

3V
三維評估框架
Viability × Value × Velocity

Viability（可行性）——技術上是否可行？六週的時間夠嗎？需要用到課程尚未教到的技術嗎？ 建議用你已經掌握的 Next.js + Supabase 堆疊，聚焦在「做出完整可用的東西」而非「嘗試最新技術」。

Value（價值性）——這個專案解決了什麼問題？ 不需要是全新的想法，「把 Notion / Airtable 的某個功能做得更好用」就是很好的切入點。 使用者不會在乎你用什麼技術，他們只在乎這個東西有沒有用。

Velocity（展示性）——完成後能清楚展示給別人看嗎？ 對面試官和潛在雇主而言，一個「能 live demo、有真實資料、視覺精美」的專案， 遠比一個技術複雜但難以展示的專案更有說服力。

推薦
個人書籤 / 閱讀清單
儲存連結、分類標籤、閱讀狀態追蹤。CRUD 完整、auth 必要、資料結構簡單清晰。
推薦
習慣追蹤器
記錄每日習慣打卡、連續天數、週報圖表。資料視覺化展示效果佳，面試亮點高。
推薦
個人記帳本
收支記錄、分類統計、月報圖表。有真實需求、功能邊界清晰、可加入預算提醒。
可行
食譜收藏平台
新增食譜、材料清單、步驟圖文、搜尋過濾。圖片上傳需整合 Supabase Storage。
可行
技能學習路線圖
自訂學習目標、拆解子任務、進度追蹤。適合做成公開分享頁面增加曝光度。
謹慎
社群 / 論壇系統
按讚、回覆、通知、追蹤——這些功能每個都有複雜的即時性需求，六週很容易做不完。

❌ 功能清單超過 20 項

❌ 依賴付費第三方 API 才能運作

❌ 先做介面，不知道資料長什麼樣

✅ MVP 功能在 3 週內能完成
✅ 資料模型不超過 5 張資料表

✅ 有圖表或視覺化（展示效果）
✅ 能用一句話解釋這個產品做什麼
03 — Feature Definition

功能定義與 MVP 範疇

MVP（Minimum Viable Product）不是「做最少的東西」，而是「用最少的功能驗證核心價值」。 用 MoSCoW 法則為所有功能排優先序，讓你清楚知道哪些是本次 Side Project 必須完成的範疇。

MV
MoSCoW 優先順序法則
Must / Should / Could / Won't

Must Have — 沒有這個功能，產品就沒有意義。這是 MVP 的核心範疇，六週必須完成。
Should Have — 重要但不是核心，若時間允許（第 16–17 週）再加入。
Could Have — 錦上添花的功能，可以做為第二版本的迭代方向。
Won't Have（this time） — 本次刻意排除，減少範疇蔓延（Scope Creep）。

範例：個人記帳本 App 功能清單

功能 說明 優先級
使用者註冊 / 登入 Email + Password，使用 Supabase Auth Must Have
新增收支記錄 金額、分類、日期、備註欄位 Must Have
收支列表 依日期排序，可依月份篩選 Must Have
編輯 / 刪除記錄 完整的 CRUD 操作 Must Have
月度支出圖表 使用 Recharts 呈現圓餅圖或長條圖 Must Have
分類管理 自訂分類名稱與顏色 Should Have
預算設定與警告 設定月預算，超出時顯示警告 Should Have
CSV 匯出 將記錄匯出為 Excel 可讀的 CSV Could Have
多帳本（家庭 / 個人） 不同帳本分開計算 Won't Have
收據 OCR 辨識 拍照自動填入金額 Won't Have
SCOPE CREEP WARNING 「範疇蔓延」是 Side Project 最大的殺手。每次你說「這個功能應該不難，順便做一下」， 就是在為自己增加不確定性。在規劃文件中白紙黑字寫下 Won't Have， 是對自己的承諾，也是避免拖延完成的紀律。
04 — User Stories

User Stories 撰寫

User Story 是從使用者視角描述功能需求的最小單位。 格式固定為：「身為一個 [角色]，我想要 [行動]，以便 [目的]。」 這個格式迫使你思考「為什麼需要這個功能」，而不只是「做什麼」。

3C
好的 User Story 三個特徵（3C）
Card × Conversation × Confirmation

Card——簡短到可以寫在一張便利貼上，通常不超過兩行。 如果需要很多字才能說清楚，代表這個 Story 太大，應該拆分。

Conversation——Story 是對話的起點，不是規格的終點。 每個 Story 後面應該有「驗收條件（Acceptance Criteria）」， 描述什麼狀況下才算「完成」。

Confirmation——驗收條件必須可以被客觀驗證， 不能是「操作起來要流暢」這種主觀描述， 而是「點擊儲存後，記錄出現在列表中且金額正確」。

範例 User Stories — 個人記帳本

訪客
身為一個訪客，我想要用 Email 和密碼註冊帳號， 以便我的記帳資料能夠安全地儲存並在不同裝置上存取。
驗收：送出表單後自動登入並導向 Dashboard；Email 已存在時顯示錯誤訊息。
Must
登入用戶
身為一個登入用戶，我想要新增一筆支出記錄（含金額、分類、日期、備註）， 以便追蹤我的每日消費。
驗收：儲存後記錄立即出現在列表中，當月支出總計即時更新。
Must
登入用戶
身為一個登入用戶，我想要查看當月各分類的支出比例圓餅圖， 以便了解自己的消費結構。
驗收：圖表依當月資料即時計算；沒有記錄時顯示空狀態提示。
Must
登入用戶
身為一個登入用戶，我想要編輯或刪除之前的記錄， 以便修正輸入錯誤。
驗收：刪除前顯示確認對話框；刪除後從列表中移除且統計數字更新。
Must
登入用戶
身為一個登入用戶，我想要設定每月預算上限， 以便在快要超支時收到警告。
驗收：支出超過預算 80% 時進度條變黃；超過 100% 時變紅並顯示警告文字。
Should
INVEST CRITERIA 好的 User Story 應符合 INVEST 原則：Independent（獨立）、Negotiable（可協商）、 Valuable（有價值）、Estimable（可估算）、Small（夠小）、Testable（可測試）。 每次寫完 Story，用這六個字過一遍，確保它是可執行的工作單位。
05 — Wireframe

Wireframe 低保真設計

Wireframe 的目的是快速驗證佈局和流程，而不是做出精美的設計稿。 在這個階段，用方塊和線條代表元素，把精力放在「使用者怎麼操作」，而不是「顏色和字型」。

低保真（Lo-Fi）
手繪或方塊圖，快速溝通佈局想法，不討論視覺細節。本週目標。
中保真（Mid-Fi）
用 Figma 的框架元件精確定義 grid、間距與元件層級，但不套用顏色。
高保真（Hi-Fi）
完整的設計稿，含顏色、字型、圖示。通常在第 16 週視覺完善階段才需要。

06 — Data Modeling

資料結構設計（ERD）

在寫任何程式碼之前，先設計好資料庫的 Schema 是最省時的做法。 Entity-Relationship Diagram（ERD） 幫助你思清楚：有哪些資料表、欄位是什麼型別、 資料表之間的關係是一對多還是多對多。

🔑
關係型資料庫核心概念
PK / FK / 一對多 / 多對多

Primary Key（PK）——每張資料表都需要唯一識別每筆記錄的欄位。 Supabase 預設使用 UUID（gen_random_uuid()），比自增 ID 更安全。

Foreign Key（FK）——用來建立表與表之間的關係。 例如 transactions 表的 user_id 欄位參照 users 表的 id， 建立「一個用戶有多筆交易記錄」的一對多關係。

多對多關係——需要一張中間表（Junction Table）來實現。 例如「一篇文章有多個標籤、一個標籤有多篇文章」，就需要 article_tags 中間表。

TABLE RELATIONSHIPS

users 1 → N categories（一個用戶有多個分類）

SQL — Supabase Schema 建立語法 
-- 啟用 UUID 擴充（Supabase 預設已啟用）
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 分類表
CREATE TABLE categories (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  color       VARCHAR(7) NOT NULL DEFAULT '#a78bfa',
  icon        TEXT,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 交易記錄表
CREATE TABLE transactions (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  category_id UUID REFERENCES categories(id) ON DELETE SET NULL,
  amount      NUMERIC(12, 2) NOT NULL CHECK (amount > 0),
  type        TEXT NOT NULL CHECK (type IN ('income', 'expense')),
  note        TEXT,
  date        DATE NOT NULL DEFAULT current_date,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- Row Level Security：每個用戶只能存取自己的資料
ALTER TABLE transactions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users own their transactions"
  ON transactions FOR ALL
  USING (auth.uid() = user_id);
RLS 很重要 Supabase 的 Row Level Security（RLS） 是資料隔離的關鍵。 忘記開啟 RLS 的話，所有登入用戶都能讀取彼此的資料。 建議在建立每張資料表後，立即設定對應的 Policy，不要留到最後才補。
07 — Tech Stack

技術選型

本課程的 Side Project 使用固定的技術堆疊，降低學習成本，聚焦在「把產品做出來」。 每個選擇都有明確的理由，而不是「因為流行」。

前端框架
Next.js 14
App Router + Server Components，同時支援 SSR 和 Client 元件。SEO 友好，Vercel 一鍵部署。
React 18
第 8–12 週已熟練，直接沿用。Next.js 的底層就是 React，過渡自然。
Tailwind CSS
第 11 週已完整學習，快速打造響應式 UI，無需在 CSS 和 JSX 之間切換。
後端 / 資料庫
Supabase
PostgreSQL 資料庫 + Auth + Storage + 即時訂閱，免費方案足以應付 Side Project 的流量。
Next.js API Routes
輕量的 BFF（Backend for Frontend），處理敏感邏輯（如 Webhook），避免 API Key 暴露在前端。
UI 元件
shadcn/ui
基於 Radix UI + Tailwind，複製原始碼到專案中，完全可自訂，無樣式侵入。
Recharts
React 原生的圖表庫，API 直觀，文件完整，適合做 Dashboard 類的資料視覺化。
開發工具
TypeScript
Supabase 可自動生成型別定義，搭配 TS 能在開發時發現大多數資料型別錯誤。
Zod
表單驗證與 API 回應的 Runtime 型別驗證，搭配 react-hook-form 使用。
Vercel
GitHub 整合後每次 push 自動部署，免費方案，Preview URL 方便展示給面試官。
為什麼不用 Redux / Zustand？ Supabase 本身就提供了資料層（資料庫）和 Auth 狀態，配合 Next.js 的 Server Components， 大多數 Side Project 不需要額外的全域狀態管理庫。 若你的應用有複雜的客戶端狀態（如多步驟表單、拖曳排序），再考慮加入 Zustand。 過早引入狀態管理反而增加複雜度。

10 — This Week's Deliverable

本週交付物

本週不寫任何功能程式碼，只產出一份規劃文件。 這份文件是後續五週開發的基礎，請認真對待。 用 Notion、Markdown 或 Google Docs 撰寫皆可。

規劃文件必要內容
01
產品簡介（一段話）
這個 Side Project 是什麼？解決誰的什麼問題？目標用戶是誰？用一到兩句話說清楚。
02
功能清單（MoSCoW 分類）
列出所有功能點，標記 Must / Should / Could / Won't，並標示哪些屬於 MVP 範疇。
03
User Stories（至少 6 則）
標準格式：「身為... 我想要... 以便...」，每則 Story 附上驗收條件（Acceptance Criteria）。
04
Wireframe（主要頁面，至少 4 頁）
手繪掃描、Excalidraw 截圖、Figma 截圖皆可。必須包含 Auth 頁、Dashboard / 首頁、列表頁、表單頁。
05
資料模型 ERD
列出所有資料表名稱、欄位（名稱 + 型別 + 必填與否）、Primary Key、Foreign Key，以及表與表的關係。
06
技術選型說明
列出使用的技術堆疊並簡要說明選擇理由。若有使用課程範圍外的套件，說明為什麼需要它。
07
週次時程規劃（14–18 週）
每週預計完成什麼里程碑，哪些功能在哪週完成。建議包含每週的「最低完成標準」作為 check-in 基準。

本週結束前完成清單

確定 Side Project 選題，能用一句話清楚說明「這是什麼、解決誰的問題」
完成 MoSCoW 功能清單，MVP 範疇（Must Have）不超過 6 個功能點
撰寫至少 6 則 User Stories，每則附驗收條件
完成至少 4 個主要頁面的 Wireframe（Auth、Dashboard、列表、表單）
設計 ERD，標明所有資料表的欄位與關係
在 Supabase 建立專案，並依 ERD 建立所有資料表（含 RLS 設定）
建立 GitHub 私有 repo，初始化 Next.js 專案並推上去
完成第 14–18 週的時程規劃，每週有明確的里程碑
PEER REVIEW 完成規劃文件後，找另一位同學互相 review。重點問題： 「這個功能的 MVP 範疇你覺得六週能完成嗎？」 「有沒有哪個功能你認為應該從 Must 移到 Should？」 旁觀者往往能看到你因為太投入而忽略的盲點。
