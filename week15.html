02 — Core Concepts

Auth 基本概念釐清

在實作之前，先把幾個經常混淆的術語釐清。 理解這些差異能幫助你在遇到問題時更快找到根因。

概念說明在 Supabase 中的對應
Authentication（認證） 確認「你是誰」——驗證身份，例如帳號密碼、生物辨識 Supabase Auth 的登入流程
Authorization（授權） 確認「你能做什麼」——驗證權限，例如 admin 才能刪除用戶 Row Level Security (RLS) Policy
JWT（JSON Web Token） 包含用戶資訊的簽名 Token，伺服器可無狀態驗證，有過期時間 Supabase 產生的 access_token（預設 1 小時）
Refresh Token 用來換取新的 access_token，壽命更長（數週） Supabase 自動管理，存在 Cookie 中
Session access_token + refresh_token + user 資訊的組合 supabase.auth.getSession() 回傳的物件
PKCE Flow OAuth 的安全驗證流程，防止授權碼攔截攻擊 Supabase 的 OAuth 預設使用 PKCE
🍪
為什麼用 Cookie，而不是 localStorage？
Next.js App Router 的 Server Component 無法存取 localStorage

localStorage 只存在於瀏覽器，Server Component 在伺服器端執行時無法讀取。 若將 Token 存在 localStorage，每次 Server Component 渲染時都無法得知用戶是否已登入， 造成 SSR 期間的身份識別問題。

HttpOnly Cookie 在每次請求時自動附帶（包含 SSR 請求），且 JavaScript 無法讀取（防 XSS）。這是現代 Web 應用儲存 Auth Token 的標準做法。 Supabase 的 @supabase/ssr 套件就是基於這個策略設計的。

Supabase Auth 初始設定

Next.js App Router 需要使用 @supabase/ssr（而不是舊版的 @supabase/auth-helpers-nextjs）， 它提供了針對 Server / Client / Middleware 三種環境分別優化的 Supabase Client 工廠函式。

src/lib/supabase/server.ts — Server Component 用 Client 
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'

export async function createClient() {
  const cookieStore = await cookies()

  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll() },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            )
          } catch {
            // Server Component 呼叫時可能拋出，可安全忽略
          }
        },
      },
    }
  )
}

04 — Signup & Login

註冊、登入與登出

Next.js App Router 推薦用 Server Actions 處理表單提交， 而不是建立 API Route 再從前端 fetch。Server Action 在伺服器端執行，能直接讀寫 Cookie， 是 Auth 操作最適合的場所。

src/app/auth/actions.ts — Server Actions 
'use server'

import { revalidatePath } from 'next/cache'
import { redirect } from 'next/navigation'
import { createClient } from '@/lib/supabase/server'

// ── 註冊 ────────────────────────────────
export async function signUp(formData: FormData) {
  const supabase = await createClient()

  const { error } = await supabase.auth.signUp({
    email: formData.get('email') as string,
    password: formData.get('password') as string,
    options: {
      emailRedirectTo: `${process.env.NEXT_PUBLIC_SITE_URL}/auth/callback`,
    },
  })

  if (error) return { error: error.message }
  redirect('/auth/verify-email')  // 提示用戶去信箱確認
}

// ── 登入 ────────────────────────────────
export async function signIn(formData: FormData) {
  const supabase = await createClient()

  const { error } = await supabase.auth.signInWithPassword({
    email: formData.get('email') as string,
    password: formData.get('password') as string,
  })

  if (error) return { error: error.message }
  revalidatePath('/', 'layout')  // 清除 cache，讓 Layout 重新讀取 Session
  redirect('/dashboard')
}

// ── 登出 ────────────────────────────────
export async function signOut() {
  const supabase = await createClient()
  await supabase.auth.signOut()
  redirect('/login')
}

// ── 忘記密碼：發送重設信 ────────────────
export async function resetPassword(formData: FormData) {
  const supabase = await createClient()

  const { error } = await supabase.auth.resetPasswordForEmail(
    formData.get('email') as string,
    { redirectTo: `${process.env.NEXT_PUBLIC_SITE_URL}/auth/update-password` }
  )

  if (error) return { error: error.message }
  return { success: true }
}

import { useState } from 'react'

            {error}

          name="email" type="email" required
          placeholder="Email"
          className="w-full px-4 py-3 bg-zinc-800 border border-zinc-700

05 — Session Management

Session 管理

在 Next.js App Router 中，讀取 Session 的方式因元件類型而不同。 搞清楚「在哪裡用哪種方式」是避免 Bug 的關鍵。

Server Component（推薦）
// app/dashboard/page.tsx（預設是 Server Component）
import { createClient } from '@/lib/supabase/server'
import { redirect } from 'next/navigation'

export default async function DashboardPage() {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  // 未登入就重定向（伺服器端，無閃爍）
  if (!user) redirect('/login')

  return <h1>Hello, {user.email}</h1>
}

// ✅ 在 Server Component 用 getUser()（不是 getSession()）
// getUser() 會向 Supabase 驗證 Token，更安全
Client Component（即時更新）
'use client'
import { useEffect, useState } from 'react'
import { createClient } from '@/lib/supabase/client'
import type { User } from '@supabase/supabase-js'

export function UserAvatar() {
  const supabase = createClient()  // 瀏覽器端，同步建立
  const [user, setUser] = useState<User | null>(null)

  useEffect(() => {
    supabase.auth.getUser().then(({ data }) => {
      setUser(data.user)
    })

    // 監聽 Auth 狀態變化（登入 / 登出）
    const { data: { subscription } } = supabase.auth.onAuthStateChange(
      (event, session) => setUser(session?.user ?? null)
    )
    return () => subscription.unsubscribe()
  }, [])

  if (!user) return null
  return <span>{user.email}</span>
}
getSession() vs getUser() 的差異 getSession() 只從 Cookie 中讀取 Session，不向 Supabase 驗證，有被偽造的風險。 getUser() 會向 Supabase 發送請求驗證 Token 有效性，是唯一安全的做法。 在需要確認身份的伺服器端邏輯（如 API Route、Server Action）中，永遠用 getUser()。
06 — Next.js Middleware

Next.js Middleware

Middleware 在請求到達任何頁面之前執行，是實作全域 Auth 保護和 自動刷新 Token的最佳位置。 它執行在 Edge Runtime，速度極快，不會阻塞頁面渲染。

middleware.ts（放在專案根目錄） 
import { createServerClient } from '@supabase/ssr'
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

// 需要登入才能存取的路徑前綴
const PROTECTED_PATHS = ['/dashboard', '/settings', '/api/protected']
// 已登入後不應存取的頁面（例如登入頁）
const AUTH_PATHS = ['/login', '/signup']

export async function middleware(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request })

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return request.cookies.getAll() },
        setAll(cookiesToSet) {
          // 同時更新 request 和 response 的 Cookie
          cookiesToSet.forEach(({ name, value }) =>
            request.cookies.set(name, value)
          )
          supabaseResponse = NextResponse.next({ request })
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          )
        },
      },
    }
  )

  // 呼叫 getUser() 自動刷新過期的 Token
  const { data: { user } } = await supabase.auth.getUser()
  const { pathname } = request.nextUrl

  // 保護路由：未登入 → 重定向到 /login
  const isProtected = PROTECTED_PATHS.some(p => pathname.startsWith(p))
  if (isProtected && !user) {
    const url = request.nextUrl.clone()
    url.pathname = '/login'
    url.searchParams.set('redirectedFrom', pathname)  // 記住原本要去哪
    return NextResponse.redirect(url)
  }

  // 已登入 → 不能再訪問 /login, /signup
  const isAuthPage = AUTH_PATHS.some(p => pathname.startsWith(p))
  if (isAuthPage && user) {
    return NextResponse.redirect(new URL('/dashboard', request.url))
  }

  return supabaseResponse
}

// 設定 Middleware 作用的路徑（排除靜態資源）
export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico|.*\\.png$).*))'],
}
redirectedFrom 參數的用途 未登入用戶被導向 /login?redirectedFrom=/dashboard 後， 登入成功時可以讀取這個參數，把用戶送回原本想去的頁面，而不是固定跳到 /dashboard。 這個細節能大幅提升 UX。在 signIn Server Action 中加入： redirect(searchParams.get('redirectedFrom') ?? '/dashboard')。
07 — Protected Routes

保護路由的完整模式

僅靠 Middleware 並不足夠——它只能做路由層級的保護。 在頁面和 API 中也需要驗證，形成縱深防禦：即使 Middleware 被繞過，資料仍然安全。

🛡
三層保護架構
Middleware → Page → Database（RLS）

第一層：Middleware——在請求進入頁面前攔截，處理重定向。速度最快，但只能做粗粒度保護。

第二層：Page / Route Handler——在頁面或 API 中再次呼叫 getUser() 驗證。 這一層能做更細緻的邏輯，例如「只有帳號擁有者才能看詳細設定頁」。

第三層：Supabase RLS——資料庫層的最後防線。 即使前兩層都被繞過，RLS Policy 確保用戶只能存取自己的資料，永遠不依賴應用層做資料隔離。

src/app/dashboard/layout.tsx — Layout 層保護 
// Layout 是 Server Component，每次渲染都會執行
import { createClient } from '@/lib/supabase/server'
import { redirect } from 'next/navigation'
import { DashboardNav } from '@/components/DashboardNav'

export default async function DashboardLayout({
  children
}: {
  children: React.ReactNode
}) {
  const supabase = await createClient()
  const { data: { user }, error } = await supabase.auth.getUser()

  // 雙重保險：Middleware 已擋一次，Layout 再擋一次
  if (error || !user) redirect('/login')

  return (
    <div className="flex min-h-screen">
      <DashboardNav user={user} />
      <main className="flex-1 p-8">{children}</main>
    </div>
  )
}
src/app/api/transactions/route.ts — API Route 保護 
import { createClient } from '@/lib/supabase/server'
import { NextResponse } from 'next/server'

export async function GET() {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  if (!user) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  // RLS 確保只能查到自己的資料（即使不寫 .eq('user_id', user.id)）
  const { data, error } = await supabase
    .from('transactions')
    .select('*, categories(name, color)')
    .order('date', { ascending: false })

  if (error) return NextResponse.json({ error: error.message }, { status: 500 })
  return NextResponse.json(data)
}
08 — Social Login / OAuth

OAuth 社群登入

Supabase 支援 Google、GitHub、Discord 等數十個 OAuth Provider。 設定只需兩個步驟：在 Provider 建立 OAuth App 取得 Client ID + Secret， 再到 Supabase Dashboard 填入——無需手動處理 PKCE 流程。

Google OAuth 設定
① Google Cloud Console → API & Services → Credentials → OAuth 2.0 Client。
② 加入 Callback URL：https://[project].supabase.co/auth/v1/callback。
③ Supabase Dashboard → Auth → Providers → Google → 填入 Client ID 和 Secret。
GitHub OAuth 設定
① GitHub → Settings → Developer Settings → OAuth Apps → New。
② Authorization callback URL 同上。
③ Supabase Dashboard → Auth → Providers → GitHub → 填入資訊。
本地開發 Callback
開發時 Callback URL 填 http://localhost:3000/auth/callback，上線後需要在 Provider 設定中額外加入生產環境的 Callback URL。
src/app/auth/actions.ts — OAuth Server Action 
export async function signInWithOAuth(provider: 'google' | 'github') {
  const supabase = await createClient()

  const { data, error } = await supabase.auth.signInWithOAuth({
    provider,
    options: {
      redirectTo: `${process.env.NEXT_PUBLIC_SITE_URL}/auth/callback`,
      queryParams: {
        access_type: 'offline',   // 取得 Refresh Token（Google 需要）
        prompt: 'consent',         // 每次都顯示授權畫面（開發時方便切換帳號）
      },
    },
  })

  if (data.url) redirect(data.url)  // 跳轉到 OAuth Provider 的授權頁
}

src/app/auth/callback/route.ts — OAuth Callback Handler 
import { createClient } from '@/lib/supabase/server'
import { NextResponse } from 'next/server'

// OAuth Provider 授權後，會帶著 code 回來這個 URL
export async function GET(request: Request) {
  const { searchParams, origin } = new URL(request.url)
  const code = searchParams.get('code')
  const next = searchParams.get('next') ?? '/dashboard'

  if (code) {
    const supabase = await createClient()
    // 用 code 換取 Session（PKCE 流程最後一步）
    const { error } = await supabase.auth.exchangeCodeForSession(code)
    if (!error) return NextResponse.redirect(`${origin}${next}`)
  }

  // 發生錯誤時跳回登入頁並附上錯誤訊息
  return NextResponse.redirect(`${origin}/login?error=auth_callback_failed`)
}

09 — Security

Auth 安全性考量

Auth 是攻擊者最常下手的地方。以下是本課程 Side Project 必須實作的安全措施， 以及常見的錯誤做法。

🔒
HttpOnly Cookie
Token 存在 HttpOnly Cookie，JavaScript 無法讀取，防止 XSS 攻擊竊取 Token。@supabase/ssr 已預設使用。
🛡
Row Level Security（RLS）
資料庫層的最後防線，確保用戶只能存取自己的資料，不依賴應用層判斷。每張資料表都要開啟。
✔
永遠用 getUser() 驗證
Server 端需要確認身份時，用 getUser() 向 Supabase 驗證，不用 getSession()（可被偽造）。
🔑
環境變數管理
Service Role Key（完整資料庫存取權）絕對不能放在前端或提交到 Git。只有 Anon Key 才可以是 NEXT_PUBLIC_。
⏱
Token 自動刷新
Middleware 中呼叫 getUser() 會自動用 Refresh Token 換取新的 Access Token，用戶不需要重新登入。
📝
輸入驗證（Zod）
Server Action 中使用 Zod 驗證所有輸入，防止惡意資料進入資料庫。不信任任何來自客戶端的資料。
🚦
Rate Limiting（限流）
Supabase 免費方案對 Auth API 有內建限流（每小時最多 30 次 Email 發送）。生產環境可透過 Supabase Dashboard 調整。
🔗
Redirect URL 白名單
在 Supabase Dashboard → Auth → URL Configuration 設定允許的 Redirect URL，防止 Open Redirect 攻擊。
絕對不要做的事 ① 把 service_role key 放在前端程式碼或 NEXT_PUBLIC_ 開頭的環境變數。
② 用 localStorage 儲存 JWT Token（XSS 可竊取）。
③ 在資料表上不開啟 RLS，靠應用層的 WHERE user_id = $user_id 做隔離（可被繞過）。
④ 用 getSession() 做身份驗證（內容可被偽造）。
