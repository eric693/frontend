核心概念 01

Optimistic Update

「樂觀更新」——不等 API 回應，直接假設操作會成功並立刻更新 UI。如果真的失敗，再回滾。這讓應用感覺比實際快了許多。

01
為什麼要 Optimistic Update
從「感覺慢」到「感覺即時」

傳統的更新流程：使用者點擊「完成」→ 等待 API 回應（可能 200–500ms）→ 畫面才更新。這個延遲在快速操作時（如一次勾選多個待辦事項）讓人沮喪。

Optimistic Update 的流程：使用者點擊「完成」→ 立刻更新 UI（假設成功）→ 同時發送 API 請求 → API 失敗時回滾 UI 並顯示錯誤。對大多數情況，API 會成功，使用者感覺操作是即時的。

實作的關鍵：在更新 UI 的同時記錄「原本的狀態」，這樣失敗時才能準確回滾。可以在 useState 中用前一個值，或在 try-catch 中捕捉失敗並 setItems 回滾。

何時不適合 Optimistic Update 金融交易、刪除重要資料（應該有確認對話框）、會影響其他使用者的操作（如搶訂位）——這些情境應該等真實回應後再更新，避免讓使用者以為操作成功但實際失敗。

function TodoList() {
  const [todos, setTodos] = useState([]);
  const { toast } = useToast();

  // ── 樂觀新增 ──
  const addTodo = async (title) => {
    const tempId = `temp-${Date.now()}`;
    const newTodo = { id: tempId, title, done: false, _optimistic: true };

    // 1. 立刻更新 UI（樂觀假設成功）
    setTodos(prev => [...prev, newTodo]);

    try {
      // 2. 發送 API 請求
      const { data, error } = await supabase
        .from('todos')
        .insert({ title })
        .select().single();

      if (error) throw error;

      // 3. 成功：把暫時 ID 換成真實 ID
      setTodos(prev => prev.map(t =>
        t.id === tempId ? { ...data, _optimistic: false } : t
      ));
    } catch (err) {
      // 4. 失敗：回滾（移除暫時加入的項目）
      setTodos(prev => prev.filter(t => t.id !== tempId));
      toast.error('新增失敗，請稍後再試');
    }
  };

  // ── 樂觀切換完成狀態 ──
  const toggleTodo = async (id, currentDone) => {
    // 1. 記錄原本狀態，立刻更新 UI
    setTodos(prev => prev.map(t =>
      t.id === id ? { ...t, done: !currentDone } : t
    ));

    try {
      const { error } = await supabase
        .from('todos')
        .update({ done: !currentDone })
        .eq('id', id);

      if (error) throw error;
    } catch (err) {
      // 失敗：回滾到原本狀態
      setTodos(prev => prev.map(t =>
        t.id === id ? { ...t, done: currentDone } : t
      ));
      toast.error('操作失敗，請稍後再試');
    }
  };

  // ── 樂觀刪除 ──
  const deleteTodo = async (id) => {
    // 記住被刪的項目，以便失敗時恢復
    const deleted = todos.find(t => t.id === id);
    setTodos(prev => prev.filter(t => t.id !== id));

    try {
      const { error } = await supabase
        .from('todos').delete().eq('id', id);
      if (error) throw error;
    } catch (err) {
      // 失敗：把刪掉的項目加回來
      setTodos(prev => [...prev, deleted].sort((a, b) =>
        new Date(a.createdAt) - new Date(b.createdAt)
      ));
      toast.error('刪除失敗');
    }
  };
}
核心概念 02

無限滾動

當列表可能有數百、數千筆資料時，一次全部載入會讓頁面卡頓。無限滾動在使用者接近底部時自動載入下一頁，讓瀏覽體驗流暢不中斷。

02
IntersectionObserver — 監測元素可見性
比 scroll 事件更高效的解法

舊方法：監聽 scroll 事件，計算 scrollTop + clientHeight >= scrollHeight。問題：scroll 事件觸發非常頻繁，即使有 throttle 也消耗效能。

IntersectionObserver：瀏覽器原生 API，監測一個「哨兵元素」（sentinel，放在列表最底部的空 div）是否進入視窗。當哨兵可見時，代表使用者到達底部，觸發載入下一頁。這個 API 是瀏覽器原生支援的，比 scroll 監聽更高效、更精確。

分頁策略：通常使用 cursor-based pagination（基於游標分頁）而非 offset（第幾頁第幾筆）。Supabase 提供 .range(from, to) 支援 offset 分頁，.lt('id', lastId) 支援游標分頁——后者在資料會即時更新的情境更準確（不會因為新資料插入而跳過或重複）。

function InfiniteList() {
  const [items, setItems]      = useState([]);
  const [loading, setLoading]  = useState(false);
  const [hasMore, setHasMore]  = useState(true);
  const [page, setPage]        = useState(0);
  const sentinelRef            = useRef(null);
  const PAGE_SIZE              = 10;

  const fetchMore = useCallback(async () => {
    if (loading || !hasMore) return;
    setLoading(true);
    try {
      const from = page * PAGE_SIZE;
      const { data, error } = await supabase
        .from('posts')
        .select('id, title, created_at')
        .order('created_at', { ascending: false })
        .range(from, from + PAGE_SIZE - 1);

      if (error) throw error;
      setItems(prev => [...prev, ...data]);
      setPage(p => p + 1);
      if (data.length < PAGE_SIZE) setHasMore(false);
    } finally {
      setLoading(false);
    }
  }, [page, loading, hasMore]);

  // 初始載入
  useEffect(() => { fetchMore(); }, []);  // eslint-disable-line

  // 設定 IntersectionObserver
  useEffect(() => {
    const sentinel = sentinelRef.current;
    if (!sentinel) return;

    const observer = new IntersectionObserver(
      entries => {
        // 哨兵元素可見時，載入下一頁
        if (entries[0].isIntersecting) fetchMore();
      },
      { threshold: 0.1 }  // 10% 可見時觸發
    );
    observer.observe(sentinel);
    return () => observer.disconnect();
  }, [fetchMore]);

  return (
    <div>
      {items.map(item => <PostCard key={item.id} item={item} />)}

      {/* 哨兵元素：放在列表最底部 */}
      <div ref={sentinelRef}>
        {loading && <Spinner />}
        {!hasMore && <p>已顯示全部內容</p>}
      </div>
    </div>
  );
}
核心概念 03

即時更新

讓多個使用者同時使用應用時，資料能即時同步——一個人新增了項目，其他人不用重新整理就能看到。Supabase Realtime 讓這件事變得簡單。

03
Supabase Realtime
WebSocket 訂閱資料庫變更

Supabase Realtime 在底層使用 WebSocket 和 PostgreSQL 的 LISTEN/NOTIFY 機制。當資料庫發生 INSERT、UPDATE、DELETE 時，所有訂閱的客戶端立刻收到通知，不需要輪詢（Polling）。

三種事件：INSERT（新資料加入）、UPDATE（資料被修改）、DELETE（資料被刪除）。可以訂閱特定資料表的所有事件，也可以用 filter 只訂閱特定條件的變更（如 user_id=eq.xxx）。

重要：使用 Realtime 時要記得在元件卸載時取消訂閱（在 useEffect 的 cleanup 函式中呼叫 channel.unsubscribe()），否則會有記憶體洩漏和重複更新的問題。

import { useEffect, useState } from 'react';
import { createClient } from '@/lib/supabase/client';

function LiveTodoList() {
  const [todos, setTodos] = useState([]);
  const supabase = createClient();

  useEffect(() => {
    // 1. 初始載入
    supabase.from('todos').select('*')
      .then(({ data }) => setTodos(data ?? []));

    // 2. 訂閱即時變更
    const channel = supabase
      .channel('todos-realtime')  // 頻道名稱（任意字串）
      .on(
        'postgres_changes',
        {
          event:  '*',         // INSERT | UPDATE | DELETE | *
          schema: 'public',
          table:  'todos',
          // filter: 'user_id=eq.xxx'  // 可選：只訂閱特定條件
        },
        (payload) => {
          const { eventType, new: newRecord, old: oldRecord } = payload;

          switch (eventType) {
            case 'INSERT':
              setTodos(prev => {
                // 避免樂觀更新後重複新增
                if (prev.some(t => t.id === newRecord.id)) return prev;
                return [...prev, newRecord];
              });
              break;
            case 'UPDATE':
              setTodos(prev => prev.map(t =>
                t.id === newRecord.id ? newRecord : t
              ));
              break;
            case 'DELETE':
              setTodos(prev => prev.filter(t => t.id !== oldRecord.id));
              break;
          }
        }
      )
      .subscribe();

    // 3. 元件卸載時取消訂閱（重要！）
    return () => {
      supabase.removeChannel(channel);
    };
  }, []);

  return <ul>{todos.map(t => <li key={t.id}>{t.title}</li>)}</ul>;
}
核心概念 04

檔案上傳

讓使用者能上傳頭像、圖片、附件——Supabase Storage 提供 S3 相容的物件儲存服務，搭配前端的拖曳上傳和進度顯示，讓上傳體驗完整且直覺。

04
Supabase Storage 架構
Bucket → 物件路徑 → 公開 URL

Supabase Storage 用 Bucket（儲存桶） 組織檔案。先在 Supabase Dashboard 建立 Bucket（如 avatars、attachments），設定公開或私有。公開 Bucket 的檔案可以用 URL 直接存取；私有 Bucket 需要生成簽名 URL。

上傳時最好用使用者 ID 作為路徑前綴：{userId}/{timestamp}-{filename}。這樣結合 Supabase Storage 的 RLS，可以確保使用者只能存取自己的檔案。

前端上傳後取得的是路徑（path），要用 supabase.storage.from('bucket').getPublicUrl(path) 取得可存取的完整 URL，再把這個 URL 存到資料庫的對應欄位。

圖片上傳前的優化 在送出前用 Canvas API 或 browser-image-compression 套件壓縮圖片。使用者上傳的照片可能是 10MB 的 RAW 檔，壓縮到 500KB 以下再上傳，可以大幅減少儲存成本和載入時間。

import { createClient } from '@/lib/supabase/client';

function AvatarUpload({ userId, onUpload }) {
  const [uploading, setUploading] = useState(false);
  const supabase = createClient();

  const handleUpload = async (file: File) => {
    // 1. 驗證檔案類型和大小
    const MAX_SIZE  = 5 * 1024 * 1024; // 5MB
    const ALLOWED   = ['image/jpeg', 'image/png', 'image/webp'];

    if (!ALLOWED.includes(file.type)) {
      throw new Error('只支援 JPG、PNG、WebP 格式');
    }
    if (file.size > MAX_SIZE) {
      throw new Error('檔案大小不能超過 5MB');
    }

    setUploading(true);
    try {
      // 2. 建立唯一路徑（避免覆蓋）
      const ext      = file.name.split('.').pop();
      const filePath = `${userId}/${Date.now()}.${ext}`;

      // 3. 上傳到 Supabase Storage
      const { error: uploadError } = await supabase.storage
        .from('avatars')
        .upload(filePath, file, {
          cacheControl: '3600',
          upsert: false,
        });
      if (uploadError) throw uploadError;

      // 4. 取得公開 URL
      const { data: { publicUrl } } = supabase.storage
        .from('avatars')
        .getPublicUrl(filePath);

      // 5. 更新資料庫中的 avatar_url
      const { error: dbError } = await supabase
        .from('profiles')
        .update({ avatar_url: publicUrl })
        .eq('id', userId);
      if (dbError) throw dbError;

      onUpload(publicUrl);  // 通知父元件更新顯示
    } finally {
      setUploading(false);
    }
  };

  // 拖曳上傳
  const handleDrop = (e) => {
    e.preventDefault();
    const file = e.dataTransfer.files[0];
    if (file) handleUpload(file);
  };

  return (
    <div
      onDrop={handleDrop}
      onDragOver={(e) => e.preventDefault()}
      className="upload-zone"
    >
      {uploading ? '上傳中...' : '拖曳或點擊上傳頭像'}
      <input type="file" onChange={e => handleUpload(e.target.files[0])} />
    </div>
  );
}
核心概念 05

搜尋與防抖（Debounce）

搜尋功能看似簡單，但做好有幾個細節：避免每次按鍵都發 API 請求、處理快速輸入後的舊請求覆蓋新結果（Race Condition）、讓使用者知道正在搜尋。

05
Debounce 與 Race Condition
搜尋的兩個核心問題

Debounce（防抖）：使用者輸入「react」要打 5 個字，不做 debounce 就會送出 5 個請求（「r」「re」「rea」「reac」「react」）。Debounce 讓你在使用者停止輸入後才送出請求（通常等 300–500ms）。

Race Condition（競態條件）：使用者快速改變搜尋詞「react」→「vue」→「angular」，三個請求同時在飛，但回應順序不一定——如果「react」的回應最慢，最終畫面會顯示「react」的結果，但輸入框裡是「angular」。解法：在每次新請求時取消上一個請求（用 AbortController）或忽略舊請求的回應（用 cleanup flag）。

Debounce vs Throttle Debounce：等使用者「停下來」才執行（適合搜尋輸入）。Throttle：每隔固定時間最多執行一次（適合 scroll 事件、resize 事件）。兩者都能減少函式的執行頻率，但策略不同。

import { useState, useEffect, useCallback } from 'react';

// ── 自訂 useDebounce Hook ──
function useDebounce<T>(value: T, delay: number = 400): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer); // 清除上一個 timer
  }, [value, delay]);
  return debounced;
}

// ── 搜尋元件 ──
function SearchPage() {
  const [query,   setQuery]   = useState('');
  const [results, setResults] = useState([]);
  const [loading, setLoading] = useState(false);

  // 只有停止輸入 400ms 後才更新
  const debouncedQuery = useDebounce(query, 400);

  useEffect(() => {
    if (!debouncedQuery.trim()) {
      setResults([]);
      return;
    }

    // AbortController 解決 Race Condition
    const controller = new AbortController();
    setLoading(true);

    supabase
      .from('posts')
      .select('id, title')
      .ilike('title', `%${debouncedQuery}%`)
      .limit(10)
      .abortSignal(controller.signal)  // Supabase 支援 AbortSignal
      .then(({ data, error }) => {
        if (error?.name === 'AbortError') return; // 被取消，忽略
        if (error) throw error;
        setResults(data ?? []);
      })
      .catch(console.error)
      .finally(() => setLoading(false));

    // cleanup：下次 effect 執行前取消上一個請求
    return () => controller.abort();

  }, [debouncedQuery]);  // 只在 debounced 值變化時觸發

  return (
    <div>
      <input
        value={query}
        onChange={e => setQuery(e.target.value)}
        placeholder="搜尋..."
      />
      {loading && <Spinner />}
      <ul>
        {results.map(r => <li key={r.id}>{r.title}</li>)}
      </ul>
    </div>
  );
}
核心概念 06

資料快取與 SWR

每次元件掛載都重新發 API 請求，不只浪費資源，還讓使用者一直看到 Loading。SWR 或 React Query 提供智慧快取，讓資料在新鮮的同時減少不必要的請求。

06
SWR — Stale-While-Revalidate
先回傳快取（舊）資料，同時在背景更新

SWR 名字來自 HTTP 快取策略 stale-while-revalidate：先立刻回傳快取中的舊資料（讓 UI 不空白），同時在背景重新取得最新資料，取到後更新 UI。使用者幾乎不需要等待 Loading。

SWR 和 React Query 都做同樣的事情，選一個即可。React Query（TanStack Query）功能更豐富，適合複雜的非同步狀態管理；SWR 更輕量，適合大多數情況。兩者都提供：自動重試、focus 時重新驗證（切回 tab 自動更新）、重複請求合併、樂觀更新支援。

使用這些庫後，你不需要再手動管理 loading/error/data 三個 state——const { data, error, isLoading } = useSWR(...) 一行搞定，而且多個元件使用同一個 key 會自動共享快取，不會重複請求。

import useSWR, { mutate } from 'swr';

// ── fetcher 函式（只需定義一次）──
const fetcher = async (url: string) => {
  const res = await fetch(url);
  if (!res.ok) throw new Error('Network error');
  return res.json();
};

// ── 基本使用 ──
function PostList() {
  const { data, error, isLoading, isValidating } = useSWR(
    '/api/posts',  // key：相同 key 的請求會共享快取
    fetcher,
    {
      revalidateOnFocus:      true,   // 切回視窗時重新驗證（預設 true）
      revalidateOnReconnect:  true,   // 斷線重連後重新驗證
      refreshInterval:        30000,  // 每 30 秒自動更新
      dedupingInterval:       2000,   // 2 秒內相同 key 只發一次請求
      fallbackData:           [],      // 沒有快取時的預設值
    }
  );
  // isLoading: 首次載入（無快取）
  // isValidating: 背景更新中（有快取但正在重驗）

  return (
    <div>
      {isLoading && <Skeleton />}
      {error && <ErrorMessage error={error} />}
      {data?.posts.map(p => <PostCard key={p.id} post={p} />)}
    </div>
  );
}

// ── 搭配樂觀更新（mutate）──
async function addPost(newPost) {
  // 樂觀更新本地快取（不等 API）
  await mutate(
    '/api/posts',
    async (current) => {
      const created = await createPost(newPost); // 真實 API
      return { posts: [...(current?.posts ?? []), created] };
    },
    {
      optimisticData: (current) => ({
        posts: [...(current?.posts ?? []), { ...newPost, id: 'temp' }],
      }),
      rollbackOnError: true,  // 失敗時自動回滾
    }
  );
}

測試你的理解

四道題目，涵蓋本週核心功能開發的關鍵概念。

QUIZ 01 / 04
實作 Optimistic Update 時，最重要的是什麼？
A先等 API 成功再更新 UI，確保資料一致
B只在成功的情況下更新 UI，失敗時什麼都不做
C更新 UI 前記錄原本狀態，API 失敗時能準確回滾
D使用 setTimeout 假裝 API 很快，再真的發請求
CORRECTOptimistic Update 的核心是「假設成功並立刻更新 UI」，但如果 API 真的失敗，你必須能夠回滾到原本狀態。所以在更新之前，要先保存一份原本的資料（如 const previous = todos.find(t => t.id === id)），這樣失敗時才能精準恢復。
TRY AGAINOptimistic Update 有兩個關鍵步驟：立刻更新 UI，以及失敗時的處理。哪個步驟需要「記錄原本狀態」？
QUIZ 02 / 04
無限滾動中，「哨兵元素」（sentinel）的作用是什麼？
A一個隱藏的計時器，每 5 秒自動載入下一頁
B放在列表底部的空元素，被 IntersectionObserver 監測，進入視窗時觸發載入
C一個特殊的 API 端點，告訴伺服器要載入哪一頁
D用來防止使用者滾動太快的限速機制
CORRECT哨兵元素（Sentinel Element）是一個放在列表最底部的空 <div>，本身不顯示任何內容。IntersectionObserver 監測它——當使用者滾動到底部，這個元素進入視窗的可見範圍，Observer 收到通知並觸發載入下一頁的函式。
TRY AGAIN「哨兵」的比喻：哨兵站在門口，當有人（使用者視窗）接近時發出信號。在無限滾動中，這個「哨兵」放在列表的哪裡？
QUIZ 03 / 04
用戶快速輸入「r」→「re」→「rea」→「reac」→「react」，不做任何處理會發送幾個搜尋 API 請求？使用 Debounce（400ms）後呢？
A5 個請求；Debounce 後還是 5 個（只是延遲）
B5 個請求；Debounce 後 0 個（全部被取消）
C1 個請求；Debounce 後 1 個（相同）
D5 個請求；Debounce 後 1 個（只發最後的「react」）
CORRECT沒有 Debounce：每次 onChange 都發請求，5 個按鍵發 5 個請求。Debounce（400ms）：每次按鍵都重置計時器，只有在停止輸入 400ms 後才發請求。如果用戶連續按鍵間隔都小於 400ms，最終只發 1 個請求（「react」）。大幅節省 API 請求次數。
TRY AGAINDebounce 的機制：每次觸發都「重置計時器」——想想連續按鍵後，最終只有哪一個按鍵能等到計時器自然倒數完？
QUIZ 04 / 04
使用 SWR 的 useSWR('/api/posts', fetcher)，在三個不同的頁面元件都使用這個 hook。會發送幾個 API 請求？
A3 個請求（每個元件各自請求）
B1 個請求（相同 key 共享快取，只發一次）
C0 個請求（SWR 永遠使用本地快取）
D取決於元件的渲染順序
CORRECTSWR 的核心能力之一：使用相同 key（'/api/posts'）的所有元件共享同一個快取和請求。即使 10 個元件都呼叫 useSWR('/api/posts')，SWR 只會發 1 個 HTTP 請求，回應後所有元件同步更新。這稱為「請求合併（Deduplication）」，是 SWR 的重要效能優化。
TRY AGAINSWR 的「S」代表 Stale（舊），「W」代表 While，「R」代表 Revalidate（重新驗證）。多個元件使用相同 key 時，SWR 如何處理？

任務：為 Side Project 加入核心功能

這週選擇 1–2 個對你的 Side Project 最重要的功能實作——不需要全部都做，選最有影響力的做好做完整。

WEEK 16 MISSION
Side Project
核心功能鍛造
把基礎架構（Auth + CRUD）轉化成真正讓使用者「哇」一聲的功能。這週的目標是讓應用感覺真正可用，而不只是能用。

完成步驟

為所有 CRUD 操作加入 Optimistic Update
審視上週完成的所有增刪改操作，逐一改為 Optimistic Update 模式：儲存原始狀態 → 立刻更新 UI → API 成功就確認，API 失敗就回滾並顯示 Toast 錯誤訊息。測試方式：把網路設為 Slow 3G，確認操作感覺仍然即時。
實作資料列表的無限滾動或分頁
如果你的應用有可能有大量資料（待辦事項、文章列表、訊息記錄），用 IntersectionObserver 實作無限滾動，或用傳統的「載入更多」按鈕分頁。確保有 Loading 指示器和「已顯示全部」的結束提示。設定每頁 10–20 筆。
加入搜尋功能（搭配 Debounce）
在主要資料列表頁加入搜尋輸入框。實作 useDebounce hook（等待 400ms 後才觸發搜尋）。在 Supabase 查詢中加入 .ilike('field', '%query%')。加入 AbortController 處理 Race Condition。顯示搜尋無結果時的 Empty State。
（選擇一）即時更新 或 檔案上傳
即時更新：用 Supabase Realtime 訂閱你的主要資料表，讓多個視窗同時開啟時資料自動同步。檔案上傳：建立 Supabase Storage Bucket，實作拖曳上傳和進度顯示，把上傳後的 URL 存到資料庫對應欄位（如頭像、附件）。
改用 SWR 或 React Query 管理資料狀態
安裝 swr 或 @tanstack/react-query，把主要的資料取得邏輯從手動的 useState + useEffect 改為 useSWR / useQuery。特別注意 loading vs isValidating 的差異——有快取時 isLoading 是 false，但仍可能在背景重驗。
整合測試：完整功能流程
在瀏覽器開兩個視窗，登入同一個帳號，在一個視窗新增/刪除/修改資料，確認另一個視窗的反應（有 Realtime 則即時同步，無則在切換回來時重新載入）。打開 Network tab，確認 Debounce 有效減少了搜尋請求次數。
加分：鍵盤快捷鍵
為主要操作加入鍵盤快捷鍵提升效率：Ctrl/Cmd+K 開啟搜尋、Ctrl/Cmd+N 新增項目、Escape 關閉 Modal。用 document.addEventListener('keydown') 實作，在 useEffect 的 cleanup 中移除監聽器。
加分：本地排序與篩選
加入在前端本地進行的排序（按日期、名稱、優先級）和篩選（完成/未完成）。這類操作不需要重新請求 API，直接用 Array.sort() 和 Array.filter() 對已載入的資料操作，並記住使用者的偏好（存在 localStorage 或 URL query params）。
