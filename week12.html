核心概念 01

Git 版本控制

Git 是目前最廣泛使用的版本控制系統。它讓你可以追蹤程式碼的每一次變更、回到任何歷史版本、與他人協作開發——是現代開發者的必備工具。

01
三個工作區域
Working Directory / Staging Area / Repository

Git 把你的檔案分成三個區域：工作目錄（Working Directory）是你實際編輯檔案的地方，修改發生在這裡。暫存區（Staging Area / Index）是一個準備區，你用 git add 把想要提交的變更放進來——這讓你可以只提交部分修改的檔案。本地倉庫（Local Repository）存放所有提交的歷史記錄，用 git commit 把暫存區的內容永久記錄下來。

理解這三層是掌握 Git 最關鍵的一步。很多人搞不清楚「為什麼 add 之後還要 commit」——因為add 只是把變更標記為「準備提交」，commit 才是真正把它存入歷史。

Git vs GitHub Git 是在你電腦上運行的版本控制軟體。GitHub 是一個儲存 Git 倉庫的雲端平台，提供協作功能。用 git push 把本地的 commit 同步到 GitHub；用 git pull 把 GitHub 的更新下載到本地。它們是不同的工具，但常常一起用。
INTERACTIVE TERMINAL點擊命令逐步執行，觀察 Git 工作流程
~/my-project $

TERMINAL — Git 完整工作流程
# ── 初始化新專案 ──
git init                     # 在當前目錄建立 .git 資料夾
git init my-project          # 建立新資料夾並初始化

# ── 設定身份（第一次用 Git 必做）──
git config --global user.name "你的名字"
git config --global user.email "你@email.com"

# ── 查看狀態 ──
git status                   # 查看哪些檔案有修改

# ── 暫存（Staging）──
git add .                    # 暫存所有修改的檔案
git add src/App.jsx          # 只暫存特定檔案
git add src/                 # 暫存整個目錄
git restore --staged file.js # 從暫存區移除（不改變檔案內容）

# ── 提交（Commit）──
git commit -m "feat: 新增使用者登入功能"
git commit -m "fix: 修正手機版導覽列樣式"
git commit -m "docs: 更新 README 安裝說明"

# ── 查看歷史 ──
git log                      # 完整歷史
git log --oneline            # 每個 commit 一行
git log --oneline --graph    # 含分支圖形

# ── 回退與還原 ──
git restore src/App.jsx      # 丟棄工作區修改（慎用！）
git reset --soft HEAD~1      # 撤銷最後一個 commit，保留修改在暫存區
git reset --mixed HEAD~1     # 撤銷 commit，修改回到工作區（預設）
git revert <commit-hash>     # 建立新 commit 反轉某個 commit（推薦用於已推送的 commit）
01
.gitignore — 忽略不需要追蹤的檔案
永遠不要提交這些東西

.gitignore 列出 Git 應該忽略（不追蹤）的檔案和目錄。這些檔案不會被 git add 加入，也不會出現在 git status 中。

最重要的三類需要忽略的東西：node_modules/（可以從 package.json 重新安裝，上傳是浪費）、.env / .env.local（包含 API 金鑰等敏感資訊，絕對不能公開）、build 產物（dist/、.next/ 等，可以重新 build）。

已提交的敏感資訊無法真正刪除 如果你不小心把 API 金鑰提交到 GitHub，即使之後用 git rm 刪除，那個 commit 的歷史記錄中還是有它。立即去 API 服務商撤銷並重新產生金鑰。這就是 .gitignore 必須在第一個 commit 前就設定好的原因。

01
Commit 訊息規範
Conventional Commits

好的 commit 訊息讓團隊協作更順暢，也讓你幾個月後看到歷史時能快速理解每個變更的意圖。業界廣泛採用 Conventional Commits 規範：type: 簡短描述（50字以內）。

常用的 type：feat（新功能）、fix（修 bug）、docs（文件）、style（格式，不影響邏輯）、refactor（重構）、test（測試）、chore（雜項，如更新依賴）。

❌ 糟糕的 commit 訊息
git commit -m "fix"
git commit -m "update"
git commit -m "asdfgh"
git commit -m "改了一些東西"
git commit -m "WIP"
# 看到這些訊息你無法知道
# 這個 commit 做了什麼
✅ 清晰的 commit 訊息
git commit -m "feat: 新增暗色模式切換功能"
git commit -m "fix: 修正 Safari 上 flex 排版問題"
git commit -m "docs: 新增 API 端點說明到 README"
git commit -m "refactor: 將 Header 拆分為子元件"
git commit -m "chore: 升級 React 至 18.3.0"
# 一眼就能理解每個變更的目的
核心概念 02

GitHub 遠端倉庫

GitHub 是全球最大的程式碼托管平台，也是你作品集的展示舞台。把程式碼推上 GitHub，不只是備份，更是讓潛在雇主和合作夥伴看見你的能力。

02
本地倉庫連接到 GitHub
remote add origin → push

流程：在 GitHub 建立空白倉庫 → 複製倉庫的 URL → 在本地用 git remote add origin <URL> 關聯 → 用 git push -u origin main 推送。-u 設定追蹤關係，之後只需 git push。

main vs master：GitHub 的預設主分支名稱現在是 main（舊版是 master）。如果你的本地是 master，用 git branch -M main 改名。

每次 git push 後，GitHub 上的程式碼就同步了。如果連接了 Vercel，這會自動觸發重新部署。這就是 CI/CD（持續整合/持續部署）的最簡單形式。

核心概念 03

Vercel 部署

Vercel 是目前最適合前端專案的部署平台——零設定支援 React、Next.js 等框架，連接 GitHub 後每次 push 自動部署，免費方案足夠個人作品集使用。

03
Vercel 部署架構
Git → Build → CDN Edge Network

Vercel 的工作流程：你 push 程式碼到 GitHub → Vercel 偵測到新 commit → 自動拉取程式碼 → 執行 npm run build → 把 build 產物部署到全球 CDN 節點 → 你的用戶從離他們最近的節點取得資源。

Vercel 的免費方案（Hobby Plan）提供：無限個人專案、自動 HTTPS、全球 CDN、自訂網域、預覽部署（每個 Pull Request 都有獨立的預覽 URL）。對個人作品集和學習用途完全足夠。

Preview 部署 Vercel 的一大亮點：當你推送到非主分支時，Vercel 會為每個 commit 建立一個獨立的預覽 URL（如 my-app-git-feat-login-yourname.vercel.app）。這讓你在合併到主分支前可以先在真實環境測試新功能，也可以把預覽連結分享給別人審查。

核心概念 04

環境變數管理

API 金鑰、資料庫密碼、第三方服務憑證——這些敏感資訊絕對不能寫在程式碼中。環境變數讓你在不同環境（本地開發/部署）使用不同的設定，同時保護敏感資訊。

04
Vite 中的環境變數
.env.local / import.meta.env

Vite 使用 .env 系列檔案管理環境變數。只有以 VITE_ 開頭的變數才會暴露給瀏覽器端程式碼（其他變數只在 Node.js 環境可見）。在程式碼中用 import.meta.env.VITE_API_KEY 存取。

檔案優先順序（高→低）：.env.local（最高優先，不應提交到 Git）→ .env.development / .env.production → .env（最低，可提交）。

前端環境變數是公開的！ VITE_ 開頭的變數會被打包進 JS bundle，任何人都能在瀏覽器 DevTools 中看到。不要把真正的私密資訊（如資料庫密碼、完整 API 密鑰）放在 VITE_ 變數。需要保護的操作應該在後端 API 處理，前端只存取公開 API。

核心概念 05

GitHub Pages 部署

GitHub Pages 是 GitHub 提供的免費靜態網站托管服務，適合部署不需要伺服器端渲染的純前端應用。設定比 Vercel 複雜一些，但完全免費且穩定。

05
GitHub Pages vs Vercel
選哪個適合你的情境

GitHub Pages：免費、直接從 GitHub 倉庫部署、URL 是 username.github.io/repo-name。限制：只能托管靜態檔案（HTML/CSS/JS），不支援 Node.js 後端、Serverless Functions。React Router 需要額外設定（因為所有路由都要回傳 index.html）。

Vercel：同樣免費（個人方案）、支援 Serverless Functions、零設定自動檢測框架、每個 PR 有預覽 URL、自訂網域更簡單。唯一缺點是不在 GitHub 的生態系中（但可以連接 GitHub）。

建議：React 應用優先選 Vercel；只有靜態 HTML/CSS 專案或需要在 GitHub 生態中展示的選 GitHub Pages。

06
在 Vercel 設定自訂網域
3 步驟設定

購買網域（Namecheap、Cloudflare、GoDaddy 等）後，在 Vercel 控制台「Settings → Domains」加入你的網域，Vercel 會給你 DNS 設定值（通常是一個 CNAME 或 A Record）。

在你的網域服務商 DNS 設定中加入這些記錄，等待 DNS 傳播（通常幾分鐘到幾小時）。Vercel 會自動為你的自訂網域申請並續約 Let's Encrypt SSL 憑證——你的網站會自動有 HTTPS，完全不需要手動操作。

免費網域選項 如果還沒準備好買網域，your-name.vercel.app 其實已經相當專業。另一個選項：.is-a.dev 提供免費的 yourname.is-a.dev 子網域給開發者（需 GitHub 申請）。

QUIZ 01 / 04
你執行 git add . 之後，這些檔案變更的狀態是？

QUIZ 02 / 04
你在 .env.local 定義了 MY_SECRET_KEY=abc123（沒有 VITE_ 前綴）。在 Vite 的 React 元件中，import.meta.env.MY_SECRET_KEY 的值是？
A"abc123"——環境變數正常被讀取
Bnull——表示未定義
Cundefined——沒有 VITE_ 前綴的變數不會暴露給瀏覽器端程式碼
D拋出錯誤——需要先在 vite.config.js 聲明
CORRECTVite 的安全機制：只有 VITE_ 開頭的環境變數才會被打包進瀏覽器端的 JS bundle。沒有前綴的變數只在 Node.js 環境（如 vite.config.js 或 Node.js 伺服器端）可見，在瀏覽器端讀取會得到 undefined。這防止了開發者不小心把伺服器端的機密暴露給用戶。

QUIZ 04 / 04
以下哪個檔案/目錄不應該被加入 .gitignore？
Anode_modules/
B.env.local
Cdist/
Dsrc/
CORRECTsrc/ 包含你的原始程式碼——這正是 Git 應該追蹤和管理的核心內容，絕對不能忽略！其他三個應該忽略：node_modules/（太大且可以重裝）、.env.local（含敏感金鑰）、dist/（build 產物，可以重新生成）。記住：Git 追蹤的是「創作」，不是「機密」和「可還原的產物」。

任務執行步驟

建立 .gitignore 並初始化 Git
在專案根目錄建立 .gitignore（內容參考概念 01 的範本），確保 node_modules/ 和 .env.local 都在裡面。然後執行 git init、git add .、git commit -m "feat: 初始化作品集專案"。用 git log --oneline 確認 commit 已成功建立。
在 GitHub 建立倉庫並推送
登入 GitHub，點擊右上角「+」→「New repository」，倉庫名稱如 my-portfolio，選 Public（方便展示），不要勾選任何初始化選項（因為本地已有 commit）。按照 GitHub 給的指令執行 git remote add origin ... 和 git push -u origin main。重整 GitHub 頁面確認程式碼上傳成功。
連接 Vercel 並完成第一次部署
前往 vercel.com，用 GitHub 帳號登入，點擊「New Project」，選擇剛建立的倉庫，Framework Preset 選 Vite，確認 Build Command 是 npm run build，Output Directory 是 dist。點擊 Deploy，等待約 1–2 分鐘，看到「Congratulations!」即成功。
設定環境變數（若有使用 API）
進入 Vercel 專案控制台 → Settings → Environment Variables，逐一加入你在 .env.local 中定義的變數（如 VITE_TMDB_API_KEY）。加入後觸發一次 Redeploy（Deployments → 點三點選單 → Redeploy），確認線上版本能正常使用 API。
測試自動部署流程
在本地修改任意一個小地方（如 Hero 標題），執行 git add . && git commit -m "feat: 更新首頁標題" && git push。前往 Vercel 控制台 Deployments 頁面，觀察部署是否自動觸發並完成。訪問你的 Vercel URL 確認修改已生效。
撰寫完整的 README.md
在專案根目錄建立 README.md，參考概念 06 的模板，至少包含：Live Demo 連結、螢幕截圖（可用 macOS 的 Cmd+Shift+4 截圖）、功能列表、技術 Stack 表格、本地安裝步驟。完成後 commit 並 push 到 GitHub，在瀏覽器確認 README 顯示正常。
建立 .env.example 供他人參考
建立 .env.example（這個可以提交到 Git），列出所有需要的環境變數但不填入實際值：VITE_API_KEY=your_api_key_here。這樣別人 clone 你的專案時知道需要設定哪些變數。在 README 的安裝步驟中加入 cp .env.example .env.local 的說明。
