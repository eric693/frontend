核心概念 01

Loading 狀態設計

使用者等待是無法避免的，但不同的 Loading UI 帶來完全不同的體驗感受。選對 Loading 模式，讓等待的焦慮降到最低。

01
Skeleton Screen vs Spinner
各有最佳適用場景

Skeleton Screen（骨架屏）：在內容載入前，顯示灰色的佔位圖形，形狀模擬真實內容的版面。優點是讓使用者知道「什麼東西即將出現在哪個位置」，大幅降低等待的焦慮感。Facebook、LinkedIn 的動態消息都用這種方式。適合頁面初始載入或大量內容的渲染。

Spinner（旋轉圖示）：通用的等待指示器，不佔太多空間。適合小範圍的操作——按鈕點擊後的處理、提交表單、上傳圖片。如果 Spinner 顯示超過 3 秒，就要考慮是否有問題。

Progress Bar（進度條）：當可以估算進度時使用——檔案上傳、多步驟流程、批次操作。比 Spinner 更讓使用者安心，因為他們知道事情正在推進。頁面頂端的細線進度條（如 YouTube、GitHub）也屬於這類，用於表示頁面或資源載入進度。

立即回饋原則 使用者點擊按鈕後，應該在 100ms 內得到視覺回饋（按鈕變灰、出現 Spinner）。如果等待超過 1 秒才有反應，使用者會以為按鈕壞了，可能重複點擊。立即回饋不代表立即完成——告訴使用者「我聽到了，正在處理」就足夠了。

function PostList() {
  const [posts, setPosts]   = useState([]);
  const [loading, setLoading] = useState(true);  // 初始為 true
  const [error, setError]   = useState(null);

  useEffect(() => {
    const fetchPosts = async () => {
      try {
        setLoading(true);
        setError(null);         // 重試時清除舊錯誤
        const res = await fetch('/api/posts');
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        const { data } = await res.json();
        setPosts(data ?? []);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);    // 無論成功失敗都關閉 loading
      }
    };
    fetchPosts();
  }, []);

  if (loading) return (
    <div>
      {[1,2,3].map(i => <PostSkeleton key={i} />)}
    </div>
  );

  if (error) return <ErrorMessage message={error} />;

  if (posts.length === 0) return <EmptyState />;

  return posts.map(post => <PostCard key={post.id} post={post} />);
}

核心概念 02

錯誤處理架構

錯誤是必然發生的，問題是如何優雅地處理它。好的錯誤處理讓使用者知道「發生了什麼」和「我能做什麼」，而不是讓他們面對一片空白。

02
錯誤的三種類型
不同錯誤需要不同的處理策略

網路錯誤（Network Error）：網路斷線、伺服器無回應（503）、逾時（Timeout）。使用者無法直接解決，但你應該提供重試按鈕，並給出「請檢查網路連線」之類的提示。

業務邏輯錯誤（Business Error）：權限不足（403）、資源不存在（404）、操作衝突（409，如用戶名稱已被使用）。這些有明確的解決方向，應該提供具體的行動指引（「這個帳號已存在，是否要改用此帳號登入？」）。

驗證錯誤（Validation Error）：表單欄位格式錯誤、必填欄位為空。應該在送出前就即時提示，或在 API 回傳 400 錯誤後顯示在對應欄位下方，不要只彈出一個通用的「錯誤」提示框。

使用者友善的錯誤訊息 不要直接把程式碼錯誤顯示給使用者：「Error: ECONNREFUSED 127.0.0.1:5432」對他們毫無意義。把技術錯誤翻譯成使用者能理解的語言：「伺服器暫時無法回應，請稍後再試。」並在 console 中保留完整的技術錯誤供開發時除錯。

// ── 統一 API 錯誤處理函式 ──
// lib/api.ts
export class ApiError extends Error {
  constructor(
    public status: number,
    message: string,
    public code?: string
  ) {
    super(message);
    this.name = 'ApiError';
  }
}

export async function apiFetch<T>(url: string, options?: RequestInit): Promise<T> {
  const res = await fetch(url, {
    headers: { 'Content-Type': 'application/json' },
    ...options,
  });
  if (!res.ok) {
    const body = await res.json().catch(() => ({}));
    throw new ApiError(res.status, body.error ?? '請求失敗', body.code);
  }
  return res.json();
}

// ── 把 status code 轉成使用者友善訊息 ──
function getUserMessage(err: unknown): string {
  if (err instanceof ApiError) {
    switch (err.status) {
      case 400: return '請求格式有誤，請確認輸入內容';
      case 401: return '請先登入後再操作';
      case 403: return '您沒有執行此操作的權限';
      case 404: return '找不到請求的資源';
      case 409: return '資料已存在，請確認後再試';
      case 429: return '請求過於頻繁，請稍後再試';
      default:  return '伺服器發生錯誤，請稍後再試';
    }
  }
  if (err instanceof TypeError) {
    return '網路連線異常，請確認網路後再試';
  }
  return '發生未知錯誤，請重新整理頁面';
}

// ── 在元件中使用 ──
try {
  const data = await apiFetch('/api/posts');
} catch (err) {
  setError(getUserMessage(err));  // 友善訊息給使用者
  console.error('API error:', err); // 技術細節給開發者
}

// ── React Error Boundary（捕捉渲染錯誤）──
import { Component, ErrorInfo, ReactNode } from 'react';

class ErrorBoundary extends Component<
  { children: ReactNode; fallback?: ReactNode },
  { hasError: boolean; error?: Error }
> {
  state = { hasError: false };

  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error('UI Error:', error, info);
    // 可以在這裡發送錯誤報告（如 Sentry）
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? <div>發生錯誤</div>;
    }
    return this.props.children;
  }
}
核心概念 03

Toast 通知系統

Toast 是不打斷使用者工作流程的輕量通知——操作完成後短暫顯示在角落，使用者不需要點擊確認就能繼續操作。比對話框（Dialog）更友善。

03
Toast 的設計原則
時機、位置、持續時間

何時用 Toast：操作成功/失敗的回饋（儲存、刪除、複製）、不需要使用者做任何決定的通知。不適合用 Toast 的情況：需要使用者確認的操作（刪除重要資料）、需要閱讀多行說明的錯誤（用 inline 錯誤訊息更好）。

位置：通常在右下角或右上角。右下角干擾較少（遠離使用者視覺焦點中心）；右上角更醒目，適合重要通知。持續時間：成功訊息 3 秒、錯誤訊息 5 秒（需要閱讀更多內容）、警告訊息 4 秒。提供手動關閉按鈕，讓不想等待的使用者自己關閉。

最多同時顯示 3 個。如果操作非常頻繁（如多個上傳同時完成），新 Toast 替換或合併舊的，不要一次堆疊十幾個。

// ── Toast Context ──
import { createContext, useContext, useState, useCallback } from 'react';

type ToastType = 'success' | 'error' | 'warn' | 'info';
interface Toast { id: string; type: ToastType; title: string; msg: string; }

const ToastContext = createContext<{
  toast: (type: ToastType, title: string, msg: string) => void;
}>({ toast: () => {} });

export function ToastProvider({ children }) {
  const [toasts, setToasts] = useState<Toast[]>([]);

  const toast = useCallback((type, title, msg) => {
    const id = Date.now().toString();
    setToasts(prev => [...prev, { id, type, title, msg }]);
    const duration = type === 'error' ? 5000 : type === 'warn' ? 4000 : 3000;
    setTimeout(() => dismiss(id), duration);
  }, []);

  const dismiss = useCallback((id: string) => {
    setToasts(prev => prev.filter(t => t.id !== id));
  }, []);

  return (
    <ToastContext.Provider value={{toast}}>
      {children}
      <div className="fixed bottom-4 right-4 flex flex-col gap-2 z-50">
        {toasts.map(t => (
          <ToastItem key={t.id} toast={t} onDismiss={dismiss} />
        ))}
      </div>
    </ToastContext.Provider>
  );
}

// ── 自訂 Hook ──
export const useToast = () => useContext(ToastContext);

// ── 在元件中使用 ──
function SaveButton() {
  const { toast } = useToast();
  const handleSave = async () => {
    try {
      await saveData();
      toast('success', '儲存成功', '您的變更已儲存');
    } catch {
      toast('error', '儲存失敗', '請稍後再試');
    }
  };
  return <button onClick={handleSave}>儲存</button>;
}
核心概念 04

Empty State 設計

當列表沒有資料時，不要只顯示「目前沒有項目」——這是一個絕佳機會，引導新使用者採取第一個行動，或讓現有使用者了解為什麼沒有看到預期的資料。

04
Empty State 的四種情境
根據情境選擇不同的設計策略

第一次使用（First Use）：使用者剛建立帳號，什麼都沒有。這是最重要的 Empty State——給他們一個明確的行動指令（「新增第一個任務」的大按鈕），讓他們立刻開始使用。

搜尋無結果（No Results）：使用者搜尋了但沒找到匹配項目。應該告訴他們可以嘗試什麼——換個關鍵字、清除篩選器、或者提示「您要新增這個嗎？」。

錯誤造成的空白（Error Empty）：資料載入失敗導致空白。這和普通的 Empty State 不同，應該告知有問題並提供重試。

有意為之的空白（Intentional Empty）：使用者主動清空了列表（刪除了所有待辦事項、清空購物車）。可以給予正向回饋（「太好了，全部完成！」）而不是消極的「沒有資料」。

interface EmptyStateProps {

function EmptyState({ icon, title, description, action, variant = 'default' }: EmptyStateProps) {

    default: 'text-gray-400',

        <button onClick={action.onClick} className="btn btn-primary">
          {action.label}
        </button>
      )}
    </div>
  );
}

核心概念 05

表單驗證與 React Hook Form

表單是使用者與應用互動最頻繁的地方。即時、清晰的驗證回饋讓使用者知道「我做對了嗎」，大幅減少錯誤率和挫折感。

05
驗證策略：何時觸發
onChange / onBlur / onSubmit

onBlur（失去焦點時）：使用者填完一個欄位離開時才驗證。比 onChange 更不打擾，是大多數欄位的推薦策略。例外：密碼強度可以在 onChange 時即時顯示（使用者還沒離開欄位就需要知道是否滿足條件）。

onChange（每次輸入時）：適合限制性較強的欄位（只能輸入數字、字元長度限制）。注意：電子郵件驗證不適合 onChange——使用者輸入到一半時「a@」就顯示錯誤很煩。

onSubmit（送出時）：最後一道防線，確保所有欄位都通過驗證才允許送出。React Hook Form 的 handleSubmit 自動在所有欄位通過驗證後才呼叫你的 submit 函式。

import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import * as z from 'zod';

// ── 1. 用 Zod 定義 Schema ──
const registerSchema = z.object({
  email: z.string()
    .min(1, '請輸入電子郵件')
    .email('請輸入有效的電子郵件格式'),
  username: z.string()
    .min(3, '使用者名稱至少 3 個字元')
    .max(20, '使用者名稱最多 20 個字元')
    .regex(/^[a-zA-Z0-9_]+$/, '只允許英文、數字和底線'),
  password: z.string()
    .min(8, '密碼至少 8 個字元')
    .regex(/[A-Z]/, '需包含至少一個大寫字母')
    .regex(/[0-9]/, '需包含至少一個數字'),
  confirmPassword: z.string(),
}).refine(
  data => data.password === data.confirmPassword,
  { message: '兩次輸入的密碼不一致', path: ['confirmPassword'] }
);

type RegisterForm = z.infer<typeof registerSchema>;

// ── 2. 在元件中使用 ──
function RegisterForm() {
  const {
    register, handleSubmit, formState: { errors, isSubmitting },
  } = useForm<RegisterForm>({
    resolver: zodResolver(registerSchema),
    mode: 'onBlur',   // 失去焦點時驗證
  });

  const onSubmit = async (data: RegisterForm) => {
    // 只有全部通過驗證才會到這裡
    await registerUser(data);
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <div>
        <input
          {...register('email')}
          className={errors.email ? 'input error' : 'input'}
        />
        {errors.email && (
          <p className="text-red-500 text-sm mt-1">
            {errors.email.message}
          </p>
        )}
      </div>

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? '處理中...' : '建立帳號'}
      </button>
    </form>
  );
}
核心概念 06

RWD 完善與收尾

RWD 不只是「縮小字體」。這週進行全面的手機版審查，修正所有影響使用者體驗的細節問題，讓應用在所有裝置上都感覺對了。

06
最常見的手機版問題
系統性檢查清單

觸控目標過小：按鈕或連結的可點擊區域太小（應至少 44×44px）。解法：增加 padding，不要只讓文字本身可點擊。min-h-[44px] min-w-[44px]（Tailwind）。

文字過小：手機上的正文字體應至少 16px，否則 iOS Safari 會自動縮放輸入框導致頁面縮放。使用 text-base（16px）作為基準，而不是 text-sm（14px）。

水平捲軸：某個元素超出視窗寬度，產生意外的橫向捲動。常見原因：固定寬度的表格、pre 標籤、min-width 過大。用 Chrome DevTools 的「Rendering → Show Layout Shift Regions」或直接在手機模式拖動視窗寬度找出問題元素。

圖片溢出：記得所有 <img> 都要有 max-width: 100%。

Chrome DevTools 測試技巧 按 Ctrl+Shift+M（Mac: Cmd+Shift+M）進入裝置模擬模式，選擇「iPhone 14 Pro」或「Samsung Galaxy S21」等具體裝置。同時開啟「Sensors」面板設定 touch 模式，才能測試真實的觸控目標大小。記得在 300%、150%、100% 三種縮放比例下各測試一遍。
常見手機版問題（點擊後截圖）
/* 按鈕太小，無法觸控 */
.btn { padding: 2px 8px; }  /* ❌ 高度 ~26px */

/* 文字過小觸發縮放 */
input { font-size: 12px; }  /* ❌ iOS 自動縮放 */

/* 圖片溢出 */
img { width: 500px; }  /* ❌ 超出容器 */

/* 表格產生水平捲軸 */
table { width: 800px; }  /* ❌ 固定寬度 */
修正後（符合規範）
/* 觸控友善按鈕 */
.btn { min-height: 44px; padding: .75rem 1.5rem; } /* ✓ */

/* 避免 iOS 縮放 */
input { font-size: 16px; }  /* ✓ >= 16px */

/* 圖片不溢出 */
img { max-width: 100%; height: auto; }  /* ✓ */

/* 表格可橫向捲動 */
.table-wrap {
  overflow-x: auto;  /* ✓ 容器可捲動 */
}

QUIZ 01 / 04
以下哪個情境最適合使用 Skeleton Screen 而非 Spinner？
A使用者點擊「刪除」按鈕，等待確認操作完成
B頁面初始載入，等待文章列表從 API 取得並渲染
C圖片上傳中，顯示上傳進度
D表單送出後，等待後端驗證回應
CORRECTSkeleton Screen 的優勢在於它告訴使用者「頁面上將會有什麼內容、在哪個位置」，特別適合頁面初始載入這類需要填充大量內容的場景。選項 A 和 D 是操作回饋（用按鈕 Loading 或小 Spinner 即可）；選項 C 有明確進度（用 Progress Bar）。
TRY AGAINSkeleton Screen 模擬的是「即將出現的內容結構」——想想哪個場景最需要讓使用者提前知道畫面上會有哪些區塊？
QUIZ 02 / 04
你的 API 回傳了 HTTP 403 錯誤。以下哪個錯誤訊息對使用者最有幫助？
A「Error: 403 Forbidden」
B「發生錯誤，請聯絡客服」
C「您沒有執行此操作的權限。如需申請權限，請聯絡管理員。」
D什麼都不顯示，直接重新導向到首頁
CORRECT好的錯誤訊息應該告訴使用者：(1) 發生了什麼、(2) 為什麼、(3) 可以怎麼辦。選項 C 做到了這三點：說明了問題（沒有權限）並提供了行動方向（聯絡管理員）。選項 A 是技術訊息，使用者看不懂；選項 B 太模糊；選項 D 讓使用者完全不知道發生了什麼。
TRY AGAIN好的錯誤訊息包含三個元素：發生了什麼 + 為什麼 + 下一步怎麼做。哪個選項包含這三個元素？
QUIZ 03 / 04
React Hook Form 的 mode: 'onBlur' 設定，表示驗證在什麼時機觸發？
A使用者每次在欄位中輸入字元時立即觸發
B使用者點擊欄位後，焦點離開（切換到其他欄位）時觸發
C使用者點擊提交按鈕送出表單時觸發
D頁面載入時自動觸發所有欄位的驗證
CORRECTonBlur 表示當使用者的焦點離開（blur）某個欄位時觸發驗證。這是大多數欄位的推薦策略，因為它不會在使用者還在輸入時就打斷他（比 onChange 更友善），同時也不需要等到送出才知道錯誤（比 onSubmit 更及時）。
TRY AGAIN「blur」在 Web 中是什麼意思？想想瀏覽器中的 focus 和 blur 事件的觸發條件。
QUIZ 04 / 04
為什麼在 iOS Safari 上，把 input 的 font-size 設為 12px 會造成問題？
AiOS Safari 不支援 12px 的字型大小，會顯示為 0px
B12px 的字型會讓中文字符顯示為方塊
CiOS Safari 會自動把 12px 的 input 轉換成 button 元件
DiOS Safari 當字型小於 16px 時，點擊 input 聚焦後會自動縮放頁面（zoom in），讓整個頁面放大
CORRECTiOS Safari 有一個保護機制：當 input 的 font-size 小於 16px 時，點擊聚焦時會自動縮放頁面（zoom in），讓字變得更容易閱讀。這讓頁面突然放大，使用者必須用手勢縮小回來，體驗很差。解法很簡單：確保所有 input/select/textarea 的 font-size 都是 16px 以上。

WEEK 17 MISSION
Side Project
UI 完善衝刺

系統性地為所有非同步操作加入 Loading 狀態，建立統一的錯誤處理，加入 Toast 通知，設計 Empty State，完善表單驗證，並全面修正 RWD 問題。

完成步驟

安裝必要的套件
執行 npm install react-hot-toast（或 sonner）安裝 Toast 套件。執行 npm install react-hook-form @hookform/resolvers zod 安裝表單驗證套件。檢查 package.json 確認安裝成功。
建立 Skeleton Screen 元件
在 components/ui/Skeleton.tsx 建立可重用的 Skeleton 元件，接受 className prop 控制形狀。為你的主要列表頁（如 TodoList、PostList）建立對應的 Skeleton 版本，在 loading 時渲染 3–5 個 Skeleton 卡片佔位。
統一所有按鈕的 Loading 狀態
建立 LoadingButton 元件，接受 loading: boolean prop，loading 時顯示 Spinner 並 disabled。把所有需要等待 API 的按鈕（新增、刪除、儲存）換成 LoadingButton，確保使用者點擊後立刻有視覺回饋，且無法重複點擊。
設定全域 Toast 並整合到所有操作
在 layout 最外層加入 <Toaster />（react-hot-toast）或 <Sonner />。找出所有 CRUD 操作，在成功時呼叫 toast.success('...')，在失敗時呼叫 toast.error('...')。確認 Toast 的措辭是使用者友善的語言。
為所有列表頁設計 Empty State
識別應用中所有可能出現空列表的位置。建立 EmptyState 元件，包含圖示、標題、描述和行動按鈕。根據情境設計：第一次使用時引導新增；搜尋無結果時提示換關鍵字；全部完成時給予正向回饋。
用 React Hook Form + Zod 重構表單
找出應用中最重要的表單（登入、新增主要資料）。用 Zod 定義驗證 Schema，用 React Hook Form 的 useForm 管理。每個欄位在 onBlur 時驗證，在欄位下方顯示即時錯誤訊息。確保 submit 按鈕在有錯誤時 disabled。
全面 RWD 審查與修正
打開 Chrome DevTools 手機模式，用 iPhone 14 Pro 尺寸逐頁瀏覽。記下所有問題：觸控目標過小、水平捲軸、文字截斷、間距過密。重點修正：所有 input font-size 至少 16px，按鈕 min-height 44px，檢查有無水平溢出元素，圖片加 max-w-full h-auto。
建立統一錯誤邊界
在 app 的頂層加入 ErrorBoundary 元件，捕捉意外的渲染錯誤。在 error.tsx（Next.js App Router）實作使用者友善的錯誤頁面，包含「重新嘗試」和「返回首頁」按鈕。用 getUserMessage(error) 函式統一把技術錯誤轉換成使用者友善的語言。
